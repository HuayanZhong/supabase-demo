# 质量验证

## 结果质量校验

任务完成后，主智能体执行以下检查：

| 检查项         | 检查方法                                            | 通过标准                                       |
| -------------- | --------------------------------------------------- | ---------------------------------------------- |
| 逻辑一致性     | 检查变更是否前后矛盾                                | 无矛盾逻辑                                     |
| 数据准确性     | 确认类型、API 参数、配置路径无误                    | 与现有文档/MCP 查询结果一致                    |
| 格式规范性     | 检查是否符合项目命名/文件结构规范（按 naming.md）   | 无不符合项                                     |
| 用户需求匹配   | 对比用户原始请求和最终输出                          | 所有需求已覆盖                                 |
| Agent 选型合规 | 对比 agent-routing.md 推荐 Agent 与实际执行的 Agent | 用推荐 Agent 或已在日志中注明降级原因          |
| 编译/类型安全  | 运行 `pnpm -F {包名} check-types`                   | 无类型错误                                     |
| 代码规范检查   | 运行 `pnpm -F {包名} lint`（如项目配置了 lint）     | 无 lint 错误；如未配置 lint 则标注"未配置"原因 |
| 已有功能未破坏 | 检查是否有静默移除已有行为                          | 未被要求的代码段未删除                         |

## 学习与优化

- 每次任务完成后记录：实际执行时间 vs 预估时间、遇到的阻塞点和解决方案、可复用的模式
- 当某类问题被重复处理 3 次以上，评估是否需要：创建新的 Skill、补充/更新规则文件、优化 Agent 选型策略
- **第三方库集成默认行为检查**：首次引入第三方库时，在初始提交中预配好优化参数（serializers、autoLogging、超时、重试策略等），避免默认值上线后才发现问题

## 纯文本回答规范

当本次任务未涉及工具调用（纯文本回答），PostToolUse 和 Stop hook 不会触发，`validate-output.ps1` 不会自动检查任务日志。此类情况下：

- 如回答涉及决策、分析或讨论，应在末尾附带一段简短的任务摘要（做了什么、发现了什么），不必输出完整结构化日志
- 如回答仅为知识性回复（如解释概念、回答问题），无需输出任何日志
