# 执行规范

## 长任务处理

**触发条件**：满足以下任一条件时，必须触发多 Agent 协同工作流程：

- 任务包含 3 个及以上可独立拆分的子任务
- 需要 2 个及以上不同领域的专业知识
- 用户主动要求并行处理

**协同流程**：

```
用户请求 → 主智能体解析并分解任务 → 按领域分派子智能体
  → 无依赖子任务并行执行，有依赖子任务按拓扑序串行
  → 主智能体整合各子任务结果 → 输出最终结果
```

**任务树输出格式**：

```
## 任务树
- P0: {关键子任务}  → 子智能体: {类型}  → 依赖: 无
- P1: {次要子任务}  → 子智能体: {类型}  → 依赖: P0 完成
- P2: {可选子任务}  → 子智能体: {类型}  → 依赖: P0, P1 完成
```

**优先级标记**：P0（阻塞性）/ P1（重要）/ P2（锦上添花），P0 优先执行，子任务粒度控制在单个 Agent 5-15 分钟可完成。

**上下文容量评估**：子任务数量 × 平均文件数。容量不足时分批执行，每批结束后输出状态摘要。

## 全面检查

**触发条件**：用户意图涉及"检查所有"、"审查全部"、"看一下所有文件"等类似表述时，必须执行以下流程：

1. 使用 Glob 列出所有相关文件（如 `**/*.service.ts`、`**/*.controller.ts`）
2. 逐一 Read 每个文件，检查以下项目：
   - 文件级注释（是否符合 comments.md）
   - 类型注解（是否有明确的返回类型）
   - JSDoc 注释（类和方法是否有 JSDoc）
   - 代码逻辑（是否有明显问题）
3. 输出检查报告，列出每个文件的状态（✅ 通过 / ⚠️ 需改进 / ❌ 有问题）
4. 按用户要求修复发现的问题

禁止只检查部分文件就声称"已全面检查"。

## Agent 选型合规

主智能体在选择 Agent 时，必须遵循 `routing.md` 的选型约束：

- 路由规则推荐了专用 Agent 的任务，**必须先调用对应 Agent** 执行核心工作
- 如认为应降级（不使用推荐 Agent），**必须先确认降级条件成立**（见 routing.md 降级条件），并在任务日志中注明
- 同一任务有多家 Agent 可选时，主智能体任选其一即可

**违规后果**：质量验证环节的"Agent 选型合规"项未通过时，主智能体必须返工。

## 需求澄清

涉及以下场景时，**必须在编码前先用 `AskUserQuestion` 确认设计方向**：

| 场景                     | 示例                                               |
| ------------------------ | -------------------------------------------------- |
| 用户流程/页面定位不明确  | 新增页面是注册前/注册后/验证后哪个环节的到达页     |
| 技术选型有多种合理方案   | 认证方式、数据存储方案、组件库选择                 |
| 需求存在两种以上合理解释 | "优化"指性能还是体验、"支持"指完整实现还是简单集成 |

**询问原则**：一次性提出（最多 3 个问题），每个问题带明确选项，阻塞性决策优先。

## 外部 API 行为验证

**涉及外部 API（第三方服务、外部 SDK、开源 API 等）时，必须先验证后使用，不得凭借训练数据猜测参数和行为。**

验证流程：

1. 使用 `WebSearch` 搜索官方文档链接
2. 使用 `WebFetch` 获取官方文档确认以下全部内容：
   - **API 端点路径**（包含完整前缀，如 `/geo/v2/city/lookup` 而非只记 `/city/lookup`）
   - **base URL**（确认 API Host 不含路径部分，路径始终从第一个 `/` 开始）
   - **请求参数**（参数名、类型、是否必需、编码方式）
   - **返回格式**（JSON 结构、错误响应格式、状态码含义）
   - **认证方式**（API Key / JWT / OAuth，header 格式）
3. 如官方文档不可用，搜索社区实践或已知的使用示例
4. 如果仍未找到可靠来源，使用 `AskUserQuestion` 询问用户确认

> 禁止行为：凭训练数据推测 API 参数、假设返回格式、猜测错误码含义、遗漏路径前缀。

## 知识边界

当遇到以下情况时，必须中止执行并向用户发起澄清询问：

| 场景                 | 询问方式                                         |
| -------------------- | ------------------------------------------------ |
| 关键信息缺失         | 说明缺什么、为什么需要、让用户提供或指定替代方案 |
| 任务目标存在多种解释 | 列出可能的解释，请用户选择                       |
| 超出当前知识库范围   | 说明知识缺口，问用户是否需要搜索或跳过           |
| 方案连续失败         | 中止当前尝试方向，先搜索验证替代方案后再继续     |
| 需要重要决策         | 列出选项及各自的利弊，让用户决定                 |
| 不确定的事实         | 先尝试搜索验证，如无结果则主动说明无法确认       |

**询问原则**：一次性提出（最多 3 个问题），每个问题提供明确选项，先处理阻塞性决策。

## 用户方向优先

当用户给出方向性反馈（如"推荐 xxx？"、"不需要 xxx"、"应该 xxx"）时，遵循以下规则：

- **直接按用户方向执行**，不要坚持自己推荐的方案，也不要花额外轮次论证用户的方案
- 用户说"算了先写出来"时，立即停止抽象/提取讨论，按最小可行方案执行
- 如果用户的方向存在技术不可行性，用 1 句话说明风险后仍按用户方向执行（除非涉及数据丢失或破坏性操作）
- 如果用户同时给出多个方向性反馈，全部接收并在同一批次中执行

> 示例：用户说"不需要提取吧，你先写出来" → AI 应当直接开始实现，而不是继续论证提取的好处。
