# 模糊需求处理

**必须在执行前使用 `AskUserQuestion` 澄清的场景：**

| 场景                       | 示例                                                              | 澄清方式                       |
| -------------------------- | ----------------------------------------------------------------- | ------------------------------ |
| **目标不明确**             | "优化一下这个页面"（性能？UI？交互？）                            | 列出可能的优化方向，让用户选择 |
| **范围不清晰**             | "加个功能"（具体什么功能？哪些页面？）                            | 询问功能的具体需求和涉及范围   |
| **技术方案有多种合理解释** | "用数据库存储"（MySQL？PostgreSQL？SQLite？）                     | 列出技术选项及各自的优缺点     |
| **涉及多个模块/文件**      | "改一下用户相关的代码"（哪些文件？Controller？Service？Entity？） | 询问具体要修改的文件和模块     |
| **需求存在矛盾**           | "既要快速实现又要完美设计"                                        | 询问优先级，让用户做取舍       |
| **缺少关键信息**           | "修复这个 bug"（但未提供错误信息或复现步骤）                      | 询问具体的错误现象和复现步骤   |

**澄清原则：**

1. **一次性提出**：最多 3 个问题，避免多轮追问
2. **提供选项**：每个问题给出 2-4 个明确选项，而非开放提问
3. **说明影响**：简要说明每个选项的优缺点或影响范围
4. **阻塞性优先**：先问影响后续执行的关键决策

**禁止猜测执行的场景：**

- 涉及 3+ 个文件的修改
- 涉及数据库结构变更
- 涉及 API 接口变更
- 涉及用户数据或权限
- 预估耗时 > 30 分钟

> 宁可多问一次，也不要做错方向。澄清成本远低于返工成本。

# 冲突解决

- 不同规则对同一事项有不同要求时，领域层 > 基础层 > 通用常识
- 同一规则文件内有冲突，以靠后的条目为准
- 主智能体无法裁决时，询问用户
