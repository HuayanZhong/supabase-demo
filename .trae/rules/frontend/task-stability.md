## 长任务分段策略

单轮对话中，同一功能反复修改超过 3 次时，说明存在稳定性问题，必须：

| #   | 步骤                                                 |
| --- | ---------------------------------------------------- |
| 1   | 暂停修改，读取当前文件完整内容                       |
| 2   | 明确输出"当前状态"（文件结构、关键 class、已知问题） |
| 3   | 对照规则逐项审查（i18n/注释/样式/布局/组件名）       |
| 4   | 一次性列出所有需要改的内容，单次提交                 |
| 5   | 修改完成后立即执行 delivery-checklist                |

> 禁止行为：修改 → 用户反馈 → 再修改 → 再反馈 → 再修改（超过 3 轮）且不重新读取文件的"打地鼠"模式。

## 多任务合并规则

同一轮响应中，满足以下条件时合并为单次批量修改：

| 条件                     | 合并策略                     |
| ------------------------ | ---------------------------- |
| 多个文件同属一个规则类别 | 1 个 Agent 批量处理          |
| 多语言 locale JSON 变更  | 单次批量操作（4 个文件一组） |
| 规则文件批量增删         | 归入同一次响应               |

## Checkpoint 规则

长任务（跨 5 轮以上）必须在以下节点输出 checkpoint：

| 节点              | 内容                                  |
| ----------------- | ------------------------------------- |
| 开始修改前        | 当前状态 + 变更计划                   |
| 每完成 3 个文件后 | 已完成 + 剩余                         |
| 全部完成后        | 最终状态 summary + delivery-checklist |

## 防退化策略

当以下信号出现时，主动切换策略：

| 信号                       | 切换                           |
| -------------------------- | ------------------------------ |
| 同一文件被连续修改 3 次    | 重新读取完整文件，一次改完     |
| 用户连续 2 次说"还是不对"  | 停下来问用户确切需求，不继续猜 |
| 同一 Agent 连续 3 轮未使用 | 下一轮强制使用该 Agent         |
