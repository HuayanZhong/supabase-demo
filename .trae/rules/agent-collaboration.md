## 0. 任务路由决策

### 0.1 意图分类

收到用户请求后，先判定所属领域：

| 领域          | 判断依据                                      | 关联规则                                              |
| ------------- | --------------------------------------------- | ----------------------------------------------------- |
| 前端开发      | 涉及 Vue 组件、Nuxt 页面、样式、i18n、UI 布局 | `nuxt.md` `styles.md` `i18n.md` `frontend/quality.md` |
| 后端开发      | 涉及 Controller、Service、Module、Entity、API | `nestjs.md` `error-handling.md` `database.md`         |
| 代码审查/质量 | 用户要求审查、审计、安全检查、测试            | `security.md` `testing.md`                            |
| 配置/跨包     | 涉及 env、依赖、monorepo 结构、构建配置       | `dependencies.md` `env-vars.md` `monorepo.md`         |
| 元任务        | 涉及 hooks 配置、规则维护、项目审计           | 直接处理                                              |

> 如无法明确归属或存在歧义 → 用 `AskUserQuestion` 澄清

### 0.2 Agent 选型

根据意图分类结果选择执行策略：

| 分类                | 执行策略         | Agent 选型                                      |
| ------------------- | ---------------- | ----------------------------------------------- |
| 前端 UI/页面        | **专属 Agent**   | `ui-designer` / `frontend-architect`            |
| 后端 API/Service/DB | **专属 Agent**   | `backend-architect`                             |
| 代码审查            | **并行多 Agent** | `code-review`(Skill) + `security-review`(Skill) |
| 全栈/跨领域         | **并行多 Agent** | `frontend-architect` + `backend-architect`      |
| 性能优化            | **专属 Agent**   | `performance-expert`                            |
| API 测试            | **专属 Agent**   | `api-test-pro`                                  |
| 合规/法律           | **专属 Agent**   | `compliance-checker`                            |
| 简单任务            | **单 Agent**     | `general_purpose_task`                          |

### 0.3 任务摘要

1. 用一句话总结用户要什么
2. 确认领域分类和 Agent 选型结果
3. 如有歧义或信息缺失 → `AskUserQuestion` 澄清

## 职责分工

### 主智能体（当前对话）

- 解析用户意图，判定任务领域与复杂度
- 按任务领域选择合适的子智能体，传递完整上下文
- 对复杂度高的任务执行分解、优先级排序和进度追踪
- 验收子智能体返回结果，执行多维度质量校验
- 遇到超出知识边界或需要用户决策时，主动发起询问
- 所有破坏性操作必须向用户确认

### 子智能体（Task 工具）

- 接收主智能体分配的子任务，按指令执行
- 任务结束时返回执行摘要（改了什么、验证了什么）
- 遇到无法解决的问题或权限不足时，立即返回主智能体
- 不访问外部网络，不执行破坏性命令
- 执行过程中记录进度状态供主智能体监控

## 子智能体选型规则

| 任务类型         | 子智能体                         | 触发条件                         |
| ---------------- | -------------------------------- | -------------------------------- |
| UI 组件/页面开发 | ui-designer / frontend-architect | 涉及 Vue 组件、模板、样式        |
| API/Service 开发 | backend-architect                | 涉及 Controller、Service、Module |
| 数据库设计       | backend-architect                | 涉及 Entity、迁移、查询          |
| E2E 测试         | api-test-pro                     | 需要浏览器自动化验证             |
| 性能优化         | performance-expert               | 涉及加载速度、渲染性能           |
| 一般编码任务     | general_purpose_task             | 非以上特定领域的代码改动         |

## 1. 长任务处理机制

### 触发条件

满足以下任一条件时，必须触发多 Agent 协同工作流程：

- 任务包含 3 个及以上可独立拆分的子任务
- 预计执行时间超过 30 分钟（涉及大量文件修改、跨领域变更）
- 需要 2 个及以上不同领域的专业知识（同时涉及前端 + 后端 + 数据库等）
- 用户主动要求并行处理

### 协同流程

```
用户请求
  │
  ├─ 1. 主智能体解析并分解任务
  │      → 输出任务树（Task Tree），标注依赖关系
  │
  ├─ 2. 按子任务领域分派子智能体
  │      → 无依赖的子任务并行执行
  │      → 有依赖的子任务按拓扑序串行
  │
  ├─ 3. 各子智能体执行并返回结果摘要
  │
  ├─ 4. 主智能体整合各子任务结果
  │      → 检查跨子任务的一致性和冲突
  │
  └─ 5. 输出整合后的最终结果
```

### 任务树输出格式

```
## 任务树
- P0: {关键子任务}  → 子智能体: {类型}  → 依赖: 无
- P1: {次要子任务}  → 子智能体: {类型}  → 依赖: P0 完成
- P2: {可选子任务}  → 子智能体: {类型}  → 依赖: P0, P1 完成
```

### 上下文容量评估

执行长任务前，主智能体需评估当前上下文是否足以完成整个任务：

- **评估依据**：任务树中子任务数量 × 每个子任务平均需读取的文件数。若上下文预计不足以承载全部子任务的完整执行过程，需在任务树输出时标注"此任务需分批执行"
- **分批策略**：先输出完整的任务树让用户确认全局计划，然后按 P0 → P1 → P2 分批执行，每批完成后输出阶段性结果，再启动下一批
- **容量不足标志**：
  - 跨 3+ 子包/模块的修改，每个子包涉及多个文件
  - 需要同时持有大量文件内容（如重构、跨层变更）才能做出正确判断
  - 子任务之间存在复杂的依赖关系，后续任务决策依赖前置任务的具体实现细节
- 分批执行时，每批结束后输出状态摘要，包含"已完成的子任务"、"下一批待执行的子任务"、"当前进度百分比"

## 2. 知识边界识别与主动询问

### 必须主动询问的场景

当系统遇到以下情况时，必须中止执行并向用户发起澄清询问：

| 场景                     | 示例                                                           | 询问方式                                         |
| ------------------------ | -------------------------------------------------------------- | ------------------------------------------------ |
| **关键信息缺失**         | 需要数据库密码但未配置、需要 API endpoint 但未找到             | 说明缺什么、为什么需要、让用户提供或指定替代方案 |
| **任务目标存在多种解释** | "优化首页"可理解为性能优化、SEO优化或视觉优化                  | 列出可能的解释，请用户选择                       |
| **超出当前知识库范围**   | 涉及未安装 skill 的框架、不熟悉的第三方 API                    | 说明知识缺口，问用户是否需要搜索或跳过           |
| **需要重要决策**         | 技术选型（用 A 方案还是 B 方案）、设计取舍（灵活性 vs 简洁性） | 列出选项及各自的利弊，让用户决定                 |
| **不确定的事实**         | API 参数、包版本、CLI 命令、配置项名                           | 先尝试搜索验证，如无结果则主动说明无法确认       |

### 询问原则

- 一次性提出多问题（最多 3 个），避免来回对话
- 每个问题提供明确的选项，尽量二选一或三选一
- 如问题超过 3 个，先处理阻塞性决策，推迟非关键问题

## 3. 文档检索与验证

### 必须调用 MCP / 搜索的场景

| 场景                   | 调用方式                                            | 要求                                 |
| ---------------------- | --------------------------------------------------- | ------------------------------------ |
| 涉及 API 用法          | `search_docs`（supabase MCP）或 `tavily_search`     | 返回来源 URL，标注版本号             |
| 需要技术参数/配置项    | `tavily_search` + `WebFetch`                        | 至少验证 2 个独立来源                |
| 用户要求提供来源依据   | `tavily_search` 或 `WebFetch`                       | 在回答中附链接                       |
| 涉及内部流程或历史任务 | 搜索 `.trae/snapshots/` 下的历史任务日志            | 引用具体日志文件                     |
| Nuxt UI 组件用法       | `nuxt-ui MCP` 的 get-component 或 search-components | 优先用 MCP，MCP 无结果再用 WebSearch |

### 验证要求

- 优先使用项目内文档（rules / skills / 日志），其次用 MCP 工具，最后用 WebSearch
- 无法验证的信息必须明确标注"此信息未验证"
- 对版本号、API 参数、配置路径等精确信息，需给出具体的引用位置
- **必须查官方文档确认 API**：代码改动涉及第三方库的 API、方法签名、配置项时，必须先打开该库的官方文档站确认当前版本的准确用法。禁止凭训练数据猜测。例：Zod → `zod.dev`，Nuxt UI → `ui.nuxt.com`，Supabase → `supabase.com/docs`

## 4. 补充控制规则

### 4a 任务分解与优先级排序

- 收到复杂任务后，先输出任务分解结构（Task Tree），标注每个子任务的领域、预估工作量、依赖关系
- 优先级标记：P0（阻塞性，必须先完成）/ P1（重要，影响交付质量）/ P2（锦上添花，可延后）
- P0 子任务优先执行，执行完毕后再推进 P1 / P2
- 子任务粒度控制在单个 Agent 可在 5-15 分钟内完成的规模

### 4b 进度监控与调整

- 多子任务并行时，主智能体追踪每个子任务的启动和完成状态
- 子任务超时（超过预期时间 2 倍未返回）→ 判断是否等待还是调整策略
- 子任务发生错误 → 主智能体评估影响范围，决定跳过/重试/回退
- 整体进度严重偏离预期时 → 主动向用户报告并提供调整方案

### 4c 结果质量校验

任务完成后，主智能体执行以下检查：

| 检查项         | 检查方法                                                | 通过标准                    |
| -------------- | ------------------------------------------------------- | --------------------------- |
| 逻辑一致性     | 检查变更是否前后矛盾                                    | 无矛盾逻辑                  |
| 数据准确性     | 确认类型、API 参数、配置路径无误                        | 与现有文档/MCP 查询结果一致 |
| 格式规范性     | 检查是否符合项目命名/文件结构规范（按 rules/naming.md） | 无不符合项                  |
| 用户需求匹配   | 对比用户原始请求和最终输出                              | 所有需求已覆盖              |
| 编译/类型安全  | 运行 `pnpm -F {包名} check-types`                       | 无类型错误                  |
| 已有功能未破坏 | 检查是否有静默移除已有行为                              | 未被要求的代码段未删除      |

### 4d 异常处理与回滚

- 文件修改前：主智能体将待修改文件的当前内容以快照形式持久化到 `.trae/snapshots/` 目录
  - 快照文件命名格式：`{timestamp}-{相对路径}.snapshot`，如 `20260707T103000-apps-frontend-pages-index.vue.snapshot`
  - 快照文件内容：修改前文件的完整原始内容
  - 快照目录已纳入 `.gitignore`，不会提交到版本库
  - 同一任务的所有快照归入以时间戳命名的子目录：`.trae/snapshots/{task-timestamp}/`
- 修改失败：子智能体返回错误详情 → 主智能体评估能否修复 → 无法修复则从 `.trae/snapshots/` 读取快照回滚
- 回滚操作完成后，清理对应的快照文件
- 严重错误（编译不通过、lint 失败）：不得提交为最终结果，必须修复后重新验证
- 错误信息记录到 `.trae/snapshots/` 对应任务子目录的 `error.json` 中，包含错误类型、影响范围、处理结果
- 如用户不满意当前结果，主智能体应回退到上一个用户确认的状态再重新执行

### 4e 学习与优化机制

- 每次任务完成后，在 `.trae/snapshots/` 中记录：
  - 实际执行时间 vs 预估时间
  - 遇到的阻塞点和解决方案
  - 可复用的模式（如某类问题适用特定子智能体 + Skill 组合）
- 当某类问题被重复处理 3 次以上，评估是否需要：
  - 创建新的 Skill 来封装经验
  - 补充/更新规则文件
  - 优化子智能体选型策略

## 冲突解决

- 不同规则对同一事项有不同要求时，领域层 > 基础层 > 通用常识
- 同一规则文件内有冲突，以靠后的条目为准
- 主智能体无法裁决时，询问用户

## 安全边界

- API Key、数据库密码必须通过 `@supabase/config` 的环境变量读取
- 禁止在源码中硬编码任何密钥
- 涉及删除文件、表结构变更、数据销毁的操作，必须先向用户确认
- **涉及文件合并、拆分、重命名、目录结构调整等破坏性结构变更，必须先提出方案经用户同意后执行**

### MCP 工具使用约束

虽然 IDE 提供了 `mcp_windows-cli`（含 `execute_command`、`ssh_execute`）、`mcp_chrome-devtools`（含 `upload_file`）、`mcp_supabase`（含 `execute_sql`）、`mcp_tavily_search`（含 `tavily_search`）、`mcp_nuxt-ui`（含 `search-components`）等工具，但子智能体调用时必须遵守以下约束：

| 工具类型                              | 允许场景                                   | 禁止场景                                 | 前置要求                            |
| ------------------------------------- | ------------------------------------------ | ---------------------------------------- | ----------------------------------- |
| `tavily_search`                       | 用户要求的网络搜索/网页内容获取            | 搜索未经授权的内网地址                   | 需用户明确指定搜索内容              |
| `search-components` / `get-component` | 用户要求的 Nuxt UI 组件查询                | —                                        | 仅用于文档查询，不修改文件          |
| `execute_command`                     | 运行用户明确要求的命令（git、pnpm、turbo） | 扫描系统目录、读取敏感文件、修改系统配置 | 命令内容必须向用户展示确认          |
| `ssh_execute`                         | 用户明确要求的远程操作                     | 连接未经用户授权的服务器                 | 连接目标必须经用户确认              |
| `upload_file`                         | 前端测试中的文件选择操作                   | 读取系统密钥文件、私密文档               | 文件路径需在项目目录内              |
| `execute_sql`                         | 用户要求的数据库查询/迁移                  | DROP TABLE/TRUNCATE 等破坏性操作         | 执行前必须展示 SQL 内容并获用户确认 |
| `evaluate_script`                     | 前端调试/测试中的 JS 评估                  | 注入恶意脚本、窃取页面数据               | 仅用于自动化测试场景                |
| `apply_migration`                     | 用户要求的数据库迁移                       | 无用户确认的自动迁移                     | 必须展示迁移文件内容后执行          |

**原则**：子智能体不直接编写 shell 脚本或批量执行未确认的命令。所有 shell 级操作必须通过 MCP 工具进行，且受上表约束。用户始终有权否决任何操作。
