# Heuristic — AI 执行阶段最佳实践

## 执行前资源加载顺序

```
1. workflows/ai/{type}.md               → 明确执行步骤
2. execution-plan/ai/（3 文件）         → 明确约束/启发/策略
3. runtime/{domain}/router.md 资源映射  → 确定可用 MCP/skill/rules
4. 按 workflow 资源表逐个加载引用文件    → 提前理解规则
```

一次性全部 Read 后再开始执行，避免执行中途频繁回头查阅。

## AI 集成工作流

### 新增 AI 功能（integrate）

```
① 读取对应 AI API 官方文档              理解接口能力和限制
② 检查项目中已有的 AI 集成模式           复用已有封装和工具函数
③ 确认 API key 和模型参数配置            从环境变量读取，不硬编码
④ 实现核心逻辑（调用/处理/返回）
⑤ 用 mock 测试验证调用链路               不依赖真实 API 端点
⑥ 集成验证后清理 mock 和测试辅助代码
```

### 修改 AI 功能（modify）

```
① Read 目标文件                         先读再改
② 如涉及新 API → 读取对应文档
③ 确认 rate limit 和错误处理是否覆盖
④ SearchReplace 修改逻辑（精准替换）
⑤ 验证修改不破坏已有功能
```

### 流式响应处理

```
① 初始化 streaming 连接                 设置超时和错误回调
② 逐块接收响应                          每块处理完后确认连接状态
③ 连接中断 → 记录断点信息               尝试重连（指数退避）
④ 完成/超时 → 关闭连接                  清理资源
```

## RAG 工作流

### 配置/修改 RAG 流程（rag）

```
① 理解 embedding pipeline              模型选择、向量维度、批处理大小
② 检查向量存储配置                      集合/索引状态、相似度算法
③ 确认文档分块策略                      块大小、重叠、分块逻辑
④ 测试检索质量                          召回率、精度、延迟
⑤ 迭代优化                              调整参数后重新测试
```

## MCP 与资源调用习惯

| 场景              | 优先工具                        | 说明                     |
| ----------------- | ------------------------------- | ------------------------ |
| 实时 web 数据搜索 | `tavily_search MCP`             | 替代通用 web search 工具 |
| web 页面内容提取  | `tavily_extract`                | 已知 URL 的正文提取      |
| 文档站点结构爬取  | `tavily_crawl`                  | 批量了解文档组织         |
| 复杂 AI 集成任务  | `ai-integration-engineer` skill | 复用已有工作流           |
| 确认 MCP 参数     | Read 工具描述文件               | 调用前必读               |
| Supabase 向量存储 | `supabase MCP → execute_sql`    | 查询向量数据             |

**原则**：tavily MCP 用于实时数据获取，skill 用于复杂任务编排，Read 用于确认工具参数。三者配合使用，不互相替代。

## 质量验证

- AI 响应在集成前必须经过质量检查：响应格式、内容合理性、错误率
- RAG 检索结果必须验证：top-k 相关性、延迟、是否返回空结果
- 流式响应必须验证：完整性和顺序正确性
- 使用 mock 测试验证 AI 调用链路，不依赖真实 API（节省成本 + 避免 rate limit）
- 集成测试仅在 mock 测试通过后运行

## 错误预防

- 所有外部 API 调用添加超时控制（connectTimeout + readTimeout）
- 流式响应添加心跳检测，识别静默断开
- 配置变更前备份当前配置
- 修改前先 Read，确认当前行号范围
- 多条命令建议用 `&&` 连接（一条失败则停止）

## 输出整理

执行完成后输出结构化摘要：

```
## 执行摘要

- 修改了哪些文件（路径 + 操作类型：新建/修改/删除）
- AI 功能变更说明（新增/修改/删除的功能点）
- 验证结果（mock 测试 / 集成测试是否通过）
- 遗留问题（未覆盖的边缘场景、待优化的性能点）
- 影响范围（哪些模块/功能会受影响）
```
