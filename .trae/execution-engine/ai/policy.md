# Policy — AI 执行阶段决策策略

## tavily_search MCP vs 通用 web 搜索决策

| 场景                         | 首选                           | 说明                                 |
| ---------------------------- | ------------------------------ | ------------------------------------ |
| 实时数据（新闻、股价、天气） | `tavily_search MCP`            | 提供结构化搜索结果                   |
| 文档/API 查阅                | `tavily_search MCP` 或专用 MCP | 优先专用 MCP（如 supabase、nuxt-ui） |
| 页面内容提取                 | `tavily_extract`               | 已知 URL，提取正文                   |
| 文档站点结构了解             | `tavily_crawl`                 | 批量爬取文档组织                     |
| 非 web 的通用信息查询        | 直接工具（如 `WebSearch`）     | 简单事实查询                         |

**原则**：需要实时 web 数据 → tavily MCP；需要 API 文档 → 对应领域 MCP；需要通用知识 → 直接工具。

## 错误分类与处理

```
阻断性错误（必须立即处理）：
├── API 调用失败（认证错误、端点不可达）
│   └── 检查 API key 配置、网络连接、端点 URL
├── API key 缺失或无效
│   └── 从环境变量重新读取，或上报配置问题
├── 类型/语法错误
│   └── 修复后继续
├── 依赖缺失（import 找不到模块）
│   └── 安装缺失依赖
└── 文件写入失败

可重试错误（记录，按策略重试）：
├── Rate limit 触发（429）
│   └── 指数退避重试：1s → 2s → 4s → 8s → max 30s
│   └── 最大重试次数：3 次
├── 网络超时（timeout）
│   └── 重试 1 次，超时时间加倍
├── 流式连接中断
│   └── 尝试重连 1 次，携带断点信息
└── 服务端 5xx 错误
    └── 重试 2 次，间隔递增

非阻断性错误（记录，完成后处理）：
├── lint 警告
├── 格式问题
├── 未使用的变量/import
└── 检索质量低于预期（RAG）
```

**原则**：API 失败立即阻断；rate limit 和超时按策略重试，不猜测重试参数；非阻断性错误容忍但记录。

## 模型选择决策（如适用）

| 场景              | 推荐模型                         | 理由             |
| ----------------- | -------------------------------- | ---------------- |
| 通用对话/文本生成 | GPT-4o / Claude 3.5 Sonnet       | 综合能力强       |
| 代码生成/补全     | Claude 3.5 Sonnet / GPT-4o       | 代码质量高       |
| 文本嵌入/RAG      | text-embedding-3-small / ada-002 | 成本低、维度适中 |
| 简单分类/提取     | GPT-4o-mini / Claude 3 Haiku     | 低成本、低延迟   |
| 流式实时响应      | 同通用模型 + streaming           | 用户体验优先     |

**决策流程**：

1. 按任务类型选择模型类别（通用/代码/嵌入）
2. 按成本/延迟约束选择具体型号
3. 通过配置（环境变量）指定模型，不在代码中硬编码

## 流式 vs 非流式响应决策

| 场景              | 推荐方式          | 理由                        |
| ----------------- | ----------------- | --------------------------- |
| 实时聊天/对话     | 流式（streaming） | 用户体验好，首 token 延迟低 |
| 代码补全/建议     | 流式（streaming） | 逐步展示结果                |
| 批量处理/后台任务 | 非流式            | 简化逻辑，减少连接管理      |
| 汇总/分析报告     | 非流式            | 需要完整上下文做后处理      |
| 长文本生成        | 流式（streaming） | 避免超时，可提前中断        |

**流式处理规则**：

- 设置合理超时（默认 30s，长文本可放宽到 60s）
- 注册连接断开和错误回调
- 每收到一个 chunk 更新心跳计数器
- 流结束后做完整性校验

**非流式处理规则**：

- 设置超时（长文本建议 120s）
- 校验响应完整性（JSON 解析、必需字段检查）
- 超时后降级为重试或返回部分结果

## 降级与回退策略

| 失败场景       | 降级方案                                                     |
| -------------- | ------------------------------------------------------------ |
| AI 服务不可用  | 返回缓存结果（如有）→ 返回友好错误提示 → 上报监控            |
| 模型响应超时   | 降级到更快模型（如 GPT-4o-mini）→ 缩短 prompt → 返回超时提示 |
| API rate limit | 排队等待（指数退避）→ 切换到备用 API key → 节流请求          |
| 流式连接中断   | 重连 1 次 → 返回已收到的部分结果 → 提示不完整                |
| RAG 检索无结果 | 降级为纯 LLM 生成 → 提示用户知识库无匹配内容                 |
| 嵌入服务不可用 | 缓存最近嵌入向量 → 降级到基于关键词的检索                    |

**原则**：降级是保底方案，所有降级路径必须在日志中记录降级原因和发生时间。降级后如果服务恢复，优先恢复到正常路径。

## 资源冲突决策

| 冲突场景                        | 策略                                   |
| ------------------------------- | -------------------------------------- |
| 多个 MCP 都可满足 web 搜索需求  | `tavily_search` 优先（专精于搜索）     |
| skill 和 MCP 信息矛盾           | MCP 优先（实时数据 > 静态知识）        |
| workflow 和 execution-plan 矛盾 | execution-plan 优先（约束 > 流程建议） |
| 用户直接指令与规则矛盾          | 用户指令优先，但标注风险               |
| 模型选择分歧（成本 vs 质量）    | 默认选质量优先；成本敏感任务选成本优先 |
