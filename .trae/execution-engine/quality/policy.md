# Policy — 质量执行阶段决策策略

## 工具选择决策：api-test-pro vs 手动 API 测试

| 场景                           | 推荐方式       | 理由                                           |
| ------------------------------ | -------------- | ---------------------------------------------- |
| 单个端点快速验证               | 手动 curl      | 无需工具加载，适合开发阶段的快速反馈           |
| 综合性端点测试（多用例）       | `api-test-pro` | 可批量运行测试用例，验证状态码+响应体+权限     |
| API 契约验证（响应结构一致性） | `api-test-pro` | 支持响应结构断言，确保接口兼容性               |
| 负载/压力测试                  | `api-test-pro` | 可配置并发和持续时间，产出 QPS/延迟/错误率指标 |
| 回归测试（每次变更后）         | `api-test-pro` | 可重复执行，集成到 CI 流程                     |
| 临时调试一个 API 的特定行为    | 手动 curl      | 灵活，适合探索性调试                           |

**原则**：探索性/临时验证用手动，正式/可重复/批量验证用 api-test-pro。

## 工具选择决策：chrome-devtools MCP vs lighthouse_audit

| 需求                        | 推荐工具                                                | 说明                                     |
| --------------------------- | ------------------------------------------------------- | ---------------------------------------- |
| 页面性能评分（FCP/LCP/TBT） | `chrome-devtools MCP → lighthouse_audit`                | 标准 Lighthouse 审计，产出可量化评分     |
| 运行时性能分析（JS 执行）   | `chrome-devtools MCP → performance_start_trace`         | Performance Trace 录制，定位长任务和卡顿 |
| 网络请求分析                | `chrome-devtools MCP → list_network_requests`           | 查看请求瀑布图、传输大小、时序           |
| 控制台错误排查              | `chrome-devtools MCP → get_console_message`             | 获取运行时错误和警告                     |
| 页面截取快照/截图           | `chrome-devtools MCP → take_snapshot / take_screenshot` | 视觉验证和 DOM 状态检查                  |

**原则**：需要综合评分 → lighthouse_audit；需要深入诊断 → 具体工具方法。

## 发现优先级

```
安全（Security）       >  最高优先级，无论严重程度必须报告
正确性（Correctness）  >  阻断性必须修复，非阻断性建议修复
性能（Performance）    >  基线对比后有显著提升再实施
风格（Style）          >  记录但不阻塞，统一修复
```

任何安全发现都排在正确性之前，即使正确性问题更明显。

## 错误分类与处理

```
阻断性错误（必须立即处理）：
├── 测试失败（任何测试未通过）
├── 类型/语法错误
├── 编译失败
├── 安全高危漏洞（CVSS ≥ 9）
├── 依赖缺失
└── 文件写入失败

非阻断性错误（记录，完成后再处理）：
├── 测试覆盖率不足（低于阈值时）
├── lint 警告
├── 格式问题
├── 风格建议
├── 性能建议（无明显收益的优化）
└── 次要安全问题（CVSS < 7）
```

非阻断性错误堆积超过 5 条时升格为优先处理。

## 阻断 vs 报告并继续决策

| 场景                               | 策略              |
| ---------------------------------- | ----------------- |
| 测试套件中有任何测试失败           | 阻断 → 修复再继续 |
| 测试覆盖率低于阈值（见下方阈值表） | 报告并继续        |
| 安全审计发现高危漏洞               | 阻断 → 立即报告   |
| 安全审计发现中/低危漏洞            | 报告并继续        |
| 代码审查发现逻辑错误               | 阻断 → 修复再继续 |
| 代码审查发现风格/注释问题          | 报告并继续        |
| 性能基线对比无明显提升             | 报告并继续        |
| 性能优化导致功能异常               | 阻断 → 回滚并修复 |
| API 测试中关键端点失败             | 阻断 → 修复再继续 |
| API 测试中非关键端点失败           | 报告并继续        |

**原则**：影响数据安全或功能正确性的必须阻断；改进性/建议性的报告即可。

## 资源冲突决策

| 冲突场景                                             | 策略                                                    |
| ---------------------------------------------------- | ------------------------------------------------------- |
| chrome-devtools MCP 和 lighthouse_audit 工具结果矛盾 | chrome-devtools MCP 优先（更精确的运行时数据）          |
| api-test-pro 和手动测试结果矛盾                      | api-test-pro 优先（可复现、可控性高）                   |
| compliance-checker 和 security-auditor 建议矛盾      | compliance-checker 优先（项目已配置的技能，一致性更高） |
| workflow 和 execution-plan 矛盾                      | execution-plan 优先（约束 > 流程建议）                  |
| 通用规则和领域规则矛盾                               | 领域规则优先（专精 > 通用）                             |
| 用户直接指令和规则矛盾                               | 用户指令优先，但标注风险                                |
| 测试数据要求影响开发环境                             | 使用独立测试数据库或 mock，不影响开发和生产环境         |

## 测试覆盖率阈值决策

| 代码类型      | 覆盖率目标 | 门禁行为                                              |
| ------------- | ---------- | ----------------------------------------------------- |
| 工具函数      | ≥ 90%      | 低于 80% 阻断，80%-90% 报告并继续                     |
| Service 层    | ≥ 80%      | 低于 70% 阻断，70%-80% 报告并继续                     |
| Controller 层 | ≥ 70%      | 低于 60% 阻断，60%-70% 报告并继续                     |
| UI 组件       | ≥ 60%      | 低于 50% 阻断（仅交互逻辑覆盖，视觉不纳入覆盖率计算） |
| 配置/类型定义 | 不要求     | 仅检查是否存在基本测试                                |

**原则**：核心逻辑（工具函数、Service）覆盖率要求最高，UI 组件只覆盖交互逻辑。覆盖率不是质量唯一指标，但低于阈值的代码不可进入生产。

## 执行顺序决策

| 情形                       | 策略                                                   |
| -------------------------- | ------------------------------------------------------ |
| 同时有测试和审查任务       | 先测试（确保证明正确），再审查（确认代码质量）         |
| 同时有安全和性能任务       | 安全优先（安全 > 性能），先完成安全审计再处理性能      |
| 同时有前端和后端质量任务   | 这不是单领域任务，暂停并上报路由分配依赖链             |
| 同一文件多次质量检查       | 每次检查后记录结果，再进入下一次检查                   |
| 测试失败和代码审查同时发生 | 先修复测试失败（阻断性），再继续审查（非阻断性可等待） |

## 质量门禁通过条件

所有以下条件满足时，方可视为质量门禁通过：

1. **全量测试通过**（`pnpm test` 零失败）
2. **无阻断性安全漏洞**（CVSS ≥ 9 的漏洞已修复）
3. **代码审查无阻断性问题**
4. **关键 API 端点测试通过**
5. **性能基线已建立**（如涉及性能优化任务）

任一条不满足则门禁不通过，不得部署。
