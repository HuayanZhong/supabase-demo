# Constraint — 质量执行阶段硬约束

## 执行前约束

- 开始执行前，必须先读取对应 `workflows/quality/{task-type}.md` 了解执行步骤
- 开始执行前，必须先读取 `execution-plan/quality/` 了解规划约束/启发/策略
- 开始执行前，必须先通过 `runtime/quality/router.md` 确认分配的资源（agents、skills、MCP）

## 资源调用约束

- 调用 MCP 前必须先通过 Read 读取对应的工具描述文件，确认参数后再调用
- 涉及安全检查时，必须使用 `compliance-checker` skill 进行敏感数据处理审查
- 涉及 API 端点测试时，必须使用 `api-test-pro` skill 进行综合性端点验证，不得仅做手动 curl 测试
- 涉及前端性能分析时，优先使用 `chrome-devtools MCP` 工具而非手动估算
- 不得直接调用未在 workflow 或路由资源表中列出的 MCP 或 skill

## 测试约束

- 完成测试任务并报告完成前，必须运行**完整测试套件**（`pnpm test`），不得仅运行新增的测试
- 新增测试不得破坏现有测试，运行全量测试通过后方可报告完成
- 测试范围必须覆盖正常路径、边界条件和错误处理三类用例
- 不得引入新的测试依赖，除非必要性经过验证

## 代码审查约束

- 代码审查必须检查项目规则（`rules/frontend/comments.md`、`rules/frontend/frontend-types.md`、`rules/frontend/styles.md`），不得仅做语法级别检查
- 审查意见必须具体到文件、行号和问题类型，不得泛泛而谈
- 审查前必须先 Read 目标代码全文，不得审查未读的代码
- 审查中发现的业务逻辑问题须记录并上报路由，不得越界修改代码

## 性能分析约束

- 提出性能优化主张前，必须建立可量化的性能基线（工具产出数据），不得凭感觉判断
- 优化实施后必须使用与基线采集时相同的工具和方法重新采集数据，提供前后对比
- 性能数据必须标注采集环境（设备、浏览器、网络条件），确保可比性

## 安全审计约束

- 安全发现**无论严重程度高低**必须报告，不得因风险低而省略
- 涉及密码、密钥、Token 的代码必须标识为需要额外审查
- 安全修复建议必须标注 CVE 编号或漏洞来源（如适用）
- 必须执行依赖漏洞扫描（`pnpm audit` 或等价工具）

## 部署约束

- **未经质量门禁通过，不得部署**
- 质量门禁包括：全量测试通过、审查无阻断性问题、安全审计无高危漏洞

## 错误处理约束

- 执行中遇到错误时，必须停止当前步骤并记录错误信息，不得尝试绕过
- 阻断性错误（测试失败、类型错误、安全高危）必须修复后才能继续
- 连续 2 次修复尝试仍失败时，必须暂停执行并上报，不得无限制重试

## 范围约束

- 质量任务只做分析、验证和测试，不做功能开发
- api-test-pro agent 只做测试验证，不得修改业务代码（不做 Edit/Write 操作）
