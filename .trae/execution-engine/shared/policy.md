# Policy — 共享包执行阶段决策策略

## 类型放置决策

| 场景                   | 策略                                         |
| ---------------------- | -------------------------------------------- |
| 单个组件用到的类型     | 组件文件内定义，不放入共享包                 |
| 单个页面用到的类型     | 页面所在目录定义，不放入共享包               |
| 两个以上模块共用       | 放入 `packages/types/` 对应模块              |
| API 请求/响应结构      | 放入 `packages/types/src/api/`（前后端共享） |
| 已有相似类型但字段不同 | 优先扩展现有类型而非新建独立类型             |
| 涉及新功能模块         | 在 `packages/types/src/` 下新建模块目录      |

## i18n key 命名约定

| 场景     | 格式                      | 示例                                            |
| -------- | ------------------------- | ----------------------------------------------- |
| 通用操作 | `common.{action}`         | `common.submit`、`common.cancel`                |
| 模块页面 | `{module}.{page}.{label}` | `auth.login.title`、`dashboard.welcome.message` |
| 表单字段 | `{module}.form.{field}`   | `profile.form.username`                         |
| 错误消息 | `{module}.error.{code}`   | `auth.error.invalid_credentials`                |
| 占位符   | 使用 `{variable}` 格式    | `welcome.back {name}`                           |

### key 结构规范

- 所有 key 使用小写 + 点号分隔
- 按功能模块分组，禁止扁平化单层结构
- 嵌套深度不超过 4 层
- 新增 key 时保持与同类 key 的命名风格一致

## Breaking Change 与向后兼容决策

| 场景             | 分类         | 策略                                   |
| ---------------- | ------------ | -------------------------------------- |
| 新增 schema/字段 | 向后兼容     | 直接新增，无需标注                     |
| 新增非必填字段   | 向后兼容     | 使用 `.optional()` 或默认值            |
| 删除字段         | **BREAKING** | 先标记 `@deprecated`，一个版本后再删除 |
| 改字段类型       | **BREAKING** | 同步更新所有引用方，标注影响范围       |
| 合并/拆分 Schema | **BREAKING** | 保留旧引用别名，逐步迁移               |
| 修改必填项为选填 | 向后兼容     | 直接修改，但需通知引用方               |
| 新增语言         | 向后兼容     | 无影响                                 |

### Breaking Change 处理流程

1. 在输出中标注 `## BREAKING CHANGE` 区块
2. 列出受影响的所有包和应用（以路径列出）
3. 对每个受影响文件说明需要做的适配
4. 如有旧类型引用，提供迁移路径示例

## 错误分类

```
阻断性错误（必须立即处理）：
├── 类型/语法错误（check-types 失败）
├── 编译失败（build 失败）
├── 依赖缺失（import 找不到模块）
├── 文件写入失败（权限/磁盘）
├── 翻译文件 key 不一致（4 语言不同步）
└── 破坏性变更未标注影响范围

非阻断性错误（记录，完成后统一处理）：
├── lint 警告（非 error 级别）
├── 格式问题（oxfmt 不一致）
├── 未使用的变量/import
└── 缺少注释说明
```

非阻断性错误超过 5 条时升格为优先处理。

## Write vs SearchReplace 决策

| 场景                        | 推荐方式        | 理由                       |
| --------------------------- | --------------- | -------------------------- |
| 新建 schema 文件            | `Write`         | 没有已有内容可参考         |
| 新建语言文件                | `Write`         | 文件内容少且全量           |
| 修改现有 schema（字段增删） | `SearchReplace` | 保留周围注释和已有字段     |
| 修改翻译值                  | `SearchReplace` | 精准匹配 key，不改其他内容 |
| 修改 lint 规则配置          | `SearchReplace` | 配置文件中精准定位         |
| 更新 index.ts re-export     | `SearchReplace` | 保持文件结构完整           |
| 文件内容少且全改            | `Write` 覆写    | 比多次替换更清晰           |

## 跨包类型引用处理

### 决策树

```
发现类型依赖关系 →
├── 同一模块内 → 直接 import，无需文档
├── types → app（单向引用）→ 正常引用，无需额外记录
├── app → types → app（跨 app 间接引用）
│   └── 在变更输出中记录影响链路
└── 包间循环引用
    └── 提取公共部分到更底层的包
```

### 记录要求

- 跨包类型依赖关系应在 `TASK_LOG` 或变更摘要中注明
- 涉及 `packages/types/` 的变更必须列出所有消费者的包路径
- 多个共享包之间的引用链应保持单向（不形成环）

## 资源冲突决策

| 冲突                            | 裁决                                   |
| ------------------------------- | -------------------------------------- |
| MCP vs Skill 信息矛盾           | MCP 优先（实时数据 > 静态知识）        |
| Workflow vs Execution Plan 矛盾 | Execution Plan 优先（约束 > 流程建议） |
| 通用 vs 共享领域规则矛盾        | 共享领域规则优先（专精 > 通用）        |
| 用户直接指令 vs 规则矛盾        | 用户指令优先，但标注风险               |
| types / i18n / lint 之间优先级  | 按依赖顺序：types → i18n → lint        |
