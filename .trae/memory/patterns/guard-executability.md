## 问题描述

守卫规则表（execution-engine/constraint.md）是高层抽象，缺少可执行细节，导致：

- 工具去重哈希算法不明确（BUG-004）
- 范围守卫对简单任务误拦截（BUG-002）
- 幂等性检查机制未定义（BUG-005）
- git 守卫干净标准未定义（BUG-006）
- 中文破坏性关键词不完整（BUG-003）

## 触发条件

当守卫规则表新增或修改时，必须同时提供以下可执行细节：

- **检测条件**：具体算法 / 关键词列表 / 检查方式
- **通过标准**：明确定义"通过"的判定
- **不通过处理**：明确动作（拦截 / 警告 / 跳过）
- **fast-path 适配**：简单任务时的降级规则

## 推荐方案

### 守卫细则段落

每个守卫项必须配有"守卫细则"段落，包含：

1. **算法/标准定义**（如哈希算法伪代码、干净标准定义）
2. **fast-path 降级规则**（无 plan 时如何处理）
3. **检查机制**（存储位置、检查方式、写入时机）
4. **与 ai-safety.md 协同关系**

### 关键词列表维护

破坏性守卫的中英文关键词列表必须定期更新：

- 英文：delete/rm/format/init/reset/drop
- 中文：删除/清理/清除/销毁/抹除/格式化/移除/还原/重置/清空
- 内容扫描：DROP TABLE/TRUNCATE

## 反模式

- 守卫规则只用抽象描述（如"核心参数语义哈希"）而不给算法
- 关键词列表不更新，遗漏常用词
- fast-path 缺失，导致简单任务被误拦截
