# 使用层面测试用例集

测试治理框架在真实使用场景下的鲁棒性——中断、打断、范围变更、恢复、降级等。

## U 类 — 用户打断与补充

| ID  | 场景           | 输入行为                             | 预期行为                               | 框架需含机制      |
| --- | -------------- | ------------------------------------ | -------------------------------------- | ----------------- |
| U1  | 执行中用户打断 | "等一下，先别执行，我改个需求"       | 当前步骤暂停，状态 checkpoint 持久化   | checkpoint + 恢复 |
| U2  | 执行中用户补充 | "这个按钮再加个图标"（已执行到一半） | 评估已完成的能否保留，追加任务到 scope | 增量追加          |
| U3  | 用户否定自己   | "算了别用表格了，改成卡片布局"       | 撤销当前方向，回退到规划阶段           | 方向变更回滚      |
| U4  | 部分接受       | "后端写完了，前端先不用了"           | 只提交已完成部分，未完成标记废弃       | 部分提交          |
| U5  | 多重打断       | 连续 3 次中断，每次改方向            | 检测到频繁变更 → 建议用户先确定需求    | 频繁变更检测      |

## V 类 — 任务异常终止

| ID  | 场景         | 输入行为                            | 预期行为                              | 框架需含机制    |
| --- | ------------ | ----------------------------------- | ------------------------------------- | --------------- |
| V1  | AI 会话超时  | 长时间无响应后重新发起              | 恢复时读取最后 checkpoint，不从头开始 | checkpoint 恢复 |
| V2  | 用户关闭 IDE | 任务执行到一半关闭再打开            | 恢复时检测工作区差异，不一致则上报    | git diff 校验   |
| V3  | MCP 中途断连 | 依赖链第 2 步时 supabase MCP 不可用 | 走降级方案或暂停，不等死              | 降级 + 暂停     |
| V4  | pnpm 冲突    | 迁移执行到一半另一个人改了同文件    | git 守卫检测到冲突，要求人工解决      | git 冲突检测    |

## W 类 — 用户引导修正

| ID  | 场景     | 输入行为                       | 预期行为                             | 框架需含机制   |
| --- | -------- | ------------------------------ | ------------------------------------ | -------------- |
| W1  | 路由纠正 | "不对，这个应该在 devops 里做" | 重新路由，已修改的回滚               | 手动重路由     |
| W2  | 质量要求 | "不需要写测试，直接部署"       | 跳过 eval 约束，走带风险的 fast-path | 约束豁免       |
| W3  | 需求收缩 | "只需要做查询，不用增删改"     | 缩减 scope，已超范围的放弃           | scope 缩减     |
| W4  | 需求膨胀 | "再加一个导出功能"             | 检测 scope 膨胀，建议新任务          | scope 膨胀检测 |

## X 类 — 复合异常

| ID  | 场景            | 输入行为                  | 预期行为                              | 框架需含机制   |
| --- | --------------- | ------------------------- | ------------------------------------- | -------------- |
| X1  | 异常 + 打断     | 迁移报错 + 用户说"先停下" | 先处理异常（回滚迁移），再 checkpoint | 异常→打断→恢复 |
| X2  | 中断 + 环境变化 | 中断期间依赖升级          | 恢复后校验环境一致性                  | 环境校验       |

---

## 测试方法

- 模拟用户输入 + 模拟执行
- 对比实际日志与预期日志
- 检查 checkpoint/恢复/回滚的正确性
- 发现的机制缺失写入 bug 并触发 evolution
