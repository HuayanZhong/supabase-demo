# Step 4: 错误模式识别

分析会话中暴露出的重复问题，输出可落地的改进行动项。

## 执行

### 4.1 重复错误检测

对照步骤 2 和 3 的输出，识别以下模式：

| 模式类型       | 识别方法                           | 示例                       |
| -------------- | ---------------------------------- | -------------------------- |
| **同类型纠正** | 同一类问题被用户纠正 ≥ 2 次        | 多次忘记输出任务日志       |
| **无声错误**   | AI 犯错但用户已无语纠正            | 重复执行已完成的审计       |
| **治理遗漏**   | hook 未触发 / 规则未引用且发生问题 | 修改后未验证               |
| **效率瓶颈**   | 同一类任务反复耗时超预期           | 每次审计都要重新读全部文件 |

### 4.2 根因分析

对每个识别出的问题，追溯根因：

```
问题描述: —
触发次数: —
首次发生: 轮次 N
末次发生: 轮次 N
根因分类: [规则缺失 / hook 配置不当 / skill 不足 / AI 能力限制 / 设计盲区]
```

### 4.3 改进建议生成

按严重度排序输出改进建议：

| 严重度   | 标准                               | 建议类型                          |
| -------- | ---------------------------------- | --------------------------------- |
| **高优** | 直接影响协作效率或治理完整性的问题 | 必须修改 → 建议修改规则/hooks     |
| **中优** | 偶发或不严重影响的问题             | 建议优化 → 建议补充规则或调整配置 |
| **低优** | 单一场景下的轻微不便               | 值得关注 → 记录待后续评估         |

改进建议必须明确以下内容：

- **建议类型**：规则增补 / hooks 调整 / skill 创建 / 文档更新 / 指令流程优化
- **建议措施**：具体修改方案（如"在 enforce-code-standards.ps1 中补充指向 X.md"）
- **预期效果**：实施后能解决什么问题
- **关联文件**：需要修改的文件路径

### 4.4 趋势对比（如有历史报告）

如果存在历史复盘报告（步骤 1 收集到），进行纵向对比：

- 上次发现的高优问题是否已解决
- 本次是否有同类问题复发（提示改进措施执行不到位）
- 治理健康状况的变化趋势（改善 / 恶化 / 持平）

## 输出摘要

```
[{步骤4 模式}] 问题 {N} 个 | 高优 {N} | 中优 {N} | 低优 {N} | 复发 {N} 个
```
