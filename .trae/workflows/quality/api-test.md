## 执行步骤

### Step 1: 确认 API 端点范围和测试用例

- 列出需要测试的 API 端点（Controller + Method）
- 阅读接口定义（Controller、Service、请求/响应类型）
- 设计测试用例：

| 用例类型 | 说明                                                             |
| -------- | ---------------------------------------------------------------- |
| 正常路径 | 标准请求，预期 2xx + 合法响应体                                  |
| 边界条件 | 空值、超长输入、非法格式、缺失必填字段                           |
| 错误处理 | 资源不存在（404）、参数错误（400）、未认证（401）、无权限（403） |

### Step 2: 使用 supabase MCP 查表结构

- 调用 `list_tables` 查看相关表
- 理解 API 背后的数据模型
- 确认测试数据是否需要预先准备

### Step 3: 执行 API 测试

验证以下维度：

- **状态码**：成功 2xx、客户端错误 4xx、服务端错误 5xx
- **响应格式**：是否符合 `{code, data, msg}` 包装格式
- **响应体**：data 字段内容是否正确、完整
- **Zod 校验**：请求体是否符合 schema 定义，错误时是否返回合理提示
- **权限控制**：未认证/无权访问是否返回正确状态码

### Step 4: 负载测试（如需要）

- 使用 k6、autocannon 或 wrk 对关键端点施压
- 关注指标：QPS、P50/P95/P99 延迟、错误率
- 确认在当前硬件条件下端点的吞吐能力

## 完成检查

- [ ] API 端点范围已确认
- [ ] 测试用例覆盖正常路径、边界、错误
- [ ] 响应格式符合 `{code, data, msg}`
- [ ] 状态码符合预期（2xx/4xx/5xx）
- [ ] 负载测试（如需要）已完成
- [ ] 测试结果已汇总
