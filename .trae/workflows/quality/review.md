# 代码审查

**对应 Agent**：compliance-checker（`agents/compliance-checker.md`）

## 触发条件

- 任务类型匹配：quality → review
- 关键词：审查、Review、审计、检查、代码规范、合规

## 准备工作

| 资源                            | 说明                                              |
| ------------------------------- | ------------------------------------------------- |
| `rules/project-architecture.md` | 项目架构概览                                      |
| `skill/turborepo`               | 构建配置                                          |
| `rules/git-commit-message.md`   | 提交信息规范                                      |
| 对应业务域的 rules              | 审查前端加载 `rules/frontend/*`，后端加载后端规则 |

### 按审查对象加载规则

| 审查对象 | 加载规则                                                                                                               |
| -------- | ---------------------------------------------------------------------------------------------------------------------- |
| 前端代码 | `rules/frontend/comments.md`、`rules/frontend/frontend-types.md`、`rules/frontend/i18n.md`、`rules/frontend/styles.md` |
| 后端代码 | 读取 `apps/backend/` 下的 Conventions 文件（如有）                                                                     |

## 执行步骤

### Step 1: 加载审查规则与上下文

- 根据审查对象（前端/后端）加载对应的 rules 文件
- 理解项目的代码风格约定和最佳实践
- 确认需要审查的文件列表和范围

### Step 2: 逐项检查

- **代码质量**：是否有死代码、冗余逻辑、复杂度超标、错误处理缺失
- **命名规范**：变量、函数、组件、文件命名是否一致且语义清晰
- **注释规范**：是否符合 `rules/frontend/comments.md`（仅前端场景）
- **类型安全**：TypeScript 类型是否完整，是否存在 `any` 滥用
- **i18n 合规**：文本是否应抽取到语言包（仅前端场景）
- **样式规范**：是否遵循 Tailwind 和 @nuxt/ui 约定（仅前端场景）
- **架构合规**：是否遵循项目分层和模块划分约定

### Step 3: 输出审查报告

报告格式：

```markdown
## 审查报告

**审查范围**：<文件列表>
**审查时间**：<日期>

### 发现的问题

| #   | 严重程度 | 文件       | 行号 | 问题描述 | 修改建议 |
| --- | -------- | ---------- | ---- | -------- | -------- |
| 1   | 中       | src/xxx.ts | 42   | ...      | ...      |

### 总结

- 问题总数：N
  - 严重：N
  - 中：N
  - 低：N
- 总体评价：<通过/有条件通过/不通过>
```

## 完成检查

- [ ] 对应业务域的 rules 已加载
- [ ] 逐项检查清单已完整执行
- [ ] 审查报告已输出

## 输出

- 代码审查报告（Markdown 格式）
- 问题列表 + 修改建议

## 约束

- **只审查不修改代码**，不做任何 Edit/Write 操作
