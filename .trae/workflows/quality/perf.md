## 执行步骤

### Step 1: 定位瓶颈 — 采集数据

根据问题领域选择合适的工具采集性能基线数据：

| 场景         | 工具                                            |
| ------------ | ----------------------------------------------- |
| 前端页面加载 | Lighthouse（FCP、LCP、TBT、CLS）                |
| 前端运行时   | 浏览器 DevTools Performance Tab                 |
| Nuxt SSR     | `nuxt build --analyze`（分析包体积）            |
| Node.js 后端 | Clinic.js（Clinic Doctor / Bubbleprof / Flame） |
| API 响应     | 应用日志 + 手动计时                             |
| 数据库查询   | Supabase MCP（慢查询日志）                      |

### Step 2: 分析 — 判断优化方向

- **前端**：
  - LCP 慢 → 首屏图片优化、减少阻塞资源、减小包体积
  - TBT 高 → 长任务拆分、代码分割、Web Worker
  - 包体积大 → Tree Shaking、动态导入、移除未使用依赖
- **后端**：
  - API 响应慢 → 数据库查询优化、缓存、N+1 检测
  - MikroORM 查询多 → populate 优化、查询字段裁剪、索引检查
  - 数据库慢 → 执行计划分析、索引缺失、连接池配置

### Step 3: 实施优化

- 遵循**最小修改原则**，只改与瓶颈直接相关的代码
- 每次只改一个优化点，避免多个变更混杂
- 不改动原有功能逻辑

### Step 4: 验证

- 使用 Step 1 中用过的工具重新采集数据
- 对比优化前后的关键指标
- 输出对比表格：

```markdown
| 指标     | 优化前 | 优化后 | 提升幅度 |
| -------- | ------ | ------ | -------- |
| LCP      | 2.8s   | 1.6s   | 43%      |
| API 响应 | 320ms  | 95ms   | 70%      |
```
