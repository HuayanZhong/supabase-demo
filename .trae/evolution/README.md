# Evolution — 元治理：治理体系自迭代

## 职责

`evolution/` 负责治理体系自身的持续改进。每次任务完成后收集经验数据，达到阈值后聚合分析，识别治理漏洞，自动改进规则。

```
evaluation/   ← 每次任务完成后写入经验数据
     │
     ▼
evolution/    ← 元治理层
  ├── 收集经验数据（每次任务完成）
  ├── 聚合分析（10次或7天）
  ├── 根因分析
  ├── 生成改进提案
  ├── 应用变更
  └── 验证效果
     │
     ▼
修改 execution-plan / execution-engine / evaluation / workflows
     │
     ▼
    下次任务按优化后的规则执行
```

## 文件结构

```
.trae/evolution/
├── README.md           # 本文档
├── constraint.md       # 元治理硬约束（证据门槛、安全不可降级）
├── heuristic.md        # 进化流程（收集 → 聚合 → 分析 → 提案 → 应用 → 验证）
└── policy.md           # 决策策略（自动/审批分界、熔断条件、优先级）
```

## 进化周期

| 阶段   | 触发条件              | 产出                 |
| ------ | --------------------- | -------------------- |
| ① 收集 | 每次任务完成后        | 经验记录             |
| ② 聚合 | 累计 10 条经验 / 7 天 | 聚合报告             |
| ③ 分析 | 聚合完成后            | 根因分析 + 模式识别  |
| ④ 提案 | 分析完成后            | 标准化治理改进提案   |
| ⑤ 应用 | 提案批准后            | 修改对应治理文件     |
| ⑥ 验证 | 应用后                | 3 次同类任务观察效果 |

---

## 核心原则

- **基于证据** — 至少 3 次相同的失败模式才触发治理变更
- **安全不可降级** — 密钥扫描、SQL注入等安全规则不可被 evolution 修改
- **不自改** — evolution 自身的变更需要人工批准
- **不自相矛盾** — 新规则不得与现有规则冲突
- **可回滚** — 每次变更记录可追溯，可恢复上一版本
