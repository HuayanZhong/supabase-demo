# Constraint — 元治理硬约束 v3

## 概述

元治理负责治理体系自身的迭代进化。本文件定义进化过程中的红线。

### 定位

```
每个任务完成 → 收集经验数据 → 达到阈值 → 生成改进提案 → 人工确认 → 应用变更
```

### 用户指导 vs AI 自主

- **用户明确要求** 的变更（如版本升级、架构调整）不在此文件的禁止修改范围内
- **AI 自主提议** 的变更才受自指保护约束
- 判断标准：用户是否在对话中明确说出"升级""改""调整"等关键词

---

## 核心硬约束

### 1. 基于证据

- 所有改进必须基于至少 1 条真实执行经验
- 不得基于假设、"最佳实践"或外部建议直接修改治理文件
- 经验数据格式参见 `heuristic.md` 的聚合报告模板

### 2. 先提案后修改

- 任何治理文件修改前必须输出提案
- 提案包含：目标文件、变更内容、变更原因、证据指向
- 人工确认后方可执行

### 3. 自指保护

以下文件禁止 AI **自主**修改（用户明确要求的变更除外）：

- `ai-safety.md`
- `evolution/constraint.md`（本文件）
- `evolution/heuristic.md`

### 4. 不可重新定义

以下规则不可被 evolution 推翻：

- 安全约束（ai-safety.md 中的禁止项）
- 语言约束（中文要求）
- 交互约束（直接回答、先读后改）

### 5. 变更粒度

- 单次 evolution 只修改 1 个治理文件
- 跨文件变更需拆分为多次提案

### 6. 回滚机制

- 每次变更前备份原文件至 `_archive/`（按 `{filename}.bak.{date}` 命名）
- 3 次任务验证后：保留 / 恢复备份 / 人工修正

### 7. 不变的部分

- `.trae/_archive/` 下的 v1 文件仅保留参考，不参与 evolution 迭代
- `.husky/pre-commit` 的修改需额外人工确认
