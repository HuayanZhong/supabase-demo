# Policy — 元治理决策策略

## 概述

元治理在有数据、有提案后，需要解决"这个改不改、怎么改、由谁批准"的问题。本文件提供决策框架。

---

## 自动化 vs 人工审批分界线

| 变更类型                 | 自动应用 | 需要审批 | 理由                     |
| ------------------------ | -------- | -------- | ------------------------ |
| 新增约束（非安全）       | ✅       | —        | 可回滚，风险可控         |
| 修改约束（非安全，放宽） | —        | ✅       | 放宽约束可能引入质量问题 |
| 修改约束（非安全，收紧） | ✅       | —        | 收紧约束是安全方向       |
| 删除约束（非安全）       | —        | ✅       | 删除有不可逆风险         |
| 新增/修改启发式          | ✅       | —        | 启发式是建议，不强制     |
| 删除启发式               | ✅       | —        | 删除建议无害             |
| 新增/修改/删除策略       | —        | ✅       | 策略影响决策链条         |
| 安全类任何变更           | —        | ✅       | 安全不可降级             |
| 新增治理文件             | —        | ✅       | 结构变更                 |
| 删除治理文件             | —        | ✅       | 结构变更                 |
| 新增骨架文件             | ✅       | —        | 骨架是建议，非强制       |
| 修改骨架内容（更新）     | ✅       | —        | 骨架应紧跟项目实际       |
| 删除骨架文件             | —        | ✅       | 删除骨架需确认不再使用   |
| 新增/修改模板约束/启发式 | ✅       | —        | 模板治理文件同启发式     |

一条速记规则：

```
约束收紧 → 自动；约束放松 → 审批
策略任何 → 审批；启发式 → 自动
安全 → 审批；新增/删除文件 → 审批
骨架更新 → 自动；骨架删除 → 审批
```

## 阈值熔断

即使满足"≥ 3 次失败"的基本阈值，以下情况也应**暂停**提案：

| 熔断条件                                                 | 说明                             | 恢复条件                       |
| -------------------------------------------------------- | -------------------------------- | ------------------------------ |
| 该类型失败最近 3 次中有 2 次是同一子任务在循环中重复产生 | 失败重复但不独立，不反映治理漏洞 | 排除循环中的重复项后重新统计   |
| 该类型失败全是同一个 task 在依赖链的多个步骤中累计的     | 单链多次失败不应算作多个独立证据 | 依赖链任务只计 1 次            |
| 提案指向的 target_file 最近 7 天内已被修改过 2 次        | 短期内反复修改说明方案不稳       | 稳定 7 天后再议                |
| 回滚过的同一个提案 30 天内再次出现                       | 已验证不可行，不应重复提案       | 有新证据（不同失败模式）可重启 |

熔断期间提案标记为 "frozen"，不执行、不丢弃。熔断条件解除后自动恢复。

## 效果评估 → 动作映射

| 验证结果                      | 后续动作                                                                         |
| ----------------------------- | -------------------------------------------------------------------------------- |
| ✅ **通过**（失败减少 ≥ 50%） | 标记已验证，记录到 changelog                                                     |
| ⚠️ **部分有效**（减少 < 50%） | 不扩大变更范围。调整现有规则再试一次。如果再验证仍不过，回滚并标记为"本方案无效" |
| ❌ **回滚**（无改善或恶化）   | 自动回滚。降低该提案 ID 优先级，30 天内不再自动提案同一方案                      |
| ❌ **引入新失败**             | 立即回滚。标记为"有害变更"，人工审查该提案的方向是否正确                         |

## 跨模式识别

单种失败类型达到阈值触发治理改进。但**多种低频失败组合出现相同根因**时也应触发。

### 识别方式

```
在聚合阶段（heuristic ②），除了按 failure_type 分组，再做一次按"根因"分组：
    ↓
检查是否有多条不同类型但指向同一个根因的问题：
```

| 看到的失败类型（不同）                     | 可能的共同根因                   |
| ------------------------------------------ | -------------------------------- |
| check-types error + lint error + test fail | 项目未配置统一的 pre-commit 钩子 |
| i18n key 遗漏 + 类型导出错误               | 前端缺乏"文件创建检查清单"指引   |
| migration 无 down + 空结果集查询           | 后端缺乏 CRUD 最佳实践的约束     |
| MCP 调用超时 + 工具返回空                  | MCP 配置或网络问题（非治理问题） |

### 跨模式动作

- 指向相同根因的复合失败 ≥ 3 种类型 → 按"高频失败"处理，触发治理改进
- 指向相同根因的复合失败 < 3 种类型 → 在聚合报告中标注"可疑根因"，但不强制提案
- 如果根因不是治理问题（如模型能力、MCP 插件稳定性）→ 输出到"非治理问题"清单，不修改治理文件

## 领域间优先级

当多个领域的治理改进提案同时出现时，执行顺序：

| 优先级 | 领域               | 理由                            |
| ------ | ------------------ | ------------------------------- |
| 1      | quality + security | 安全和质量是底线                |
| 2      | shared             | 共享包影响所有其他领域          |
| 3      | backend            | 后端是数据和服务的基础          |
| 4      | frontend           | 前端依赖后端，可稍后            |
| 5      | devops             | 基础设施变更的节奏慢于业务代码  |
| 6      | ai                 | AI 功能通常依赖于已有后端和前端 |

同领域多个提案时：

- P0 优先（安全类）→ P1 优先（高频失败）→ P2（治理遗漏）→ P3（优化建议）
- 先修"缺了什么"（constraint），再修"怎么更好"（heuristic）

## 模板更新策略

### 更新类型判断

| 检测结果                        | 动作                       | 优先级 |
| ------------------------------- | -------------------------- | ------ |
| 骨架结构与实际代码有结构性差异  | 更新骨架内容               | P1     |
| 某类文件频繁出现但无对应骨架    | 新增骨架文件               | P2     |
| 骨架引用的 API 已弃用           | 立即更新骨架               | P0     |
| 骨架长期未被引用（10 次无使用） | 标记为废弃，人工确认后删除 | P3     |

### 更新方式

- 骨架内容更新：自动应用，在 changelog 中记录
- 新增骨架文件：自动创建，在 changelog 中记录
- 删除骨架文件：需人工审批
- 骨架更新无需验证期（骨架不影响执行结果，只影响产出格式）
- 但如果同一个骨架连续更新 3 次，说明骨架不够稳定，暂停自动更新并输出报告

## 治理覆盖度评分

元治理应定期评估治理体系的"覆盖度"，识别薄弱领域。

### 评分方式

```
每完成一次聚合后，计算各领域的"治理覆盖度评分"：
```

| 评分维度     | 权重 | 计算方式                                          |
| ------------ | ---- | ------------------------------------------------- |
| 通过率       | 30%  | 该领域通过任务数 / 该领域总任务数                 |
| 首次通过率   | 25%  | 该领域首次就通过的任务数 / 总任务数               |
| 失败覆盖度   | 25%  | 该领域失败类型中被治理覆盖的类型数 / 总失败类型数 |
| 循环收敛效率 | 20%  | 该领域平均循环次数（越低越好）                    |

### 评分等级

| 评分  | 等级        | 动作                                     |
| ----- | ----------- | ---------------------------------------- |
| ≥ 90  | A（优秀）   | 维持当前治理，关注变化趋势               |
| 70–89 | B（良好）   | 可优化，优先处理高频失败                 |
| 50–69 | C（需改进） | 该领域治理存在明显缺口，优先分配进化资源 |
| < 50  | D（薄弱）   | 该领域治理严重不足，建议人工介入         |

评分建议每 2 次聚合评估一次（约 20 个任务），不过度频繁。

## 竞合策略

### 同时多个提案修改同一文件

```
同一 target_file 有多个提案：
    ↓
按优先级排序（安全 > 高频 > 遗漏 > 优化）
    ↓
应用最高优先级的提案
    ↓
处于"待验证"期时，其他修改该文件的提案暂缓
    ↓
验证通过后，评估下一个提案是否需要修改同一位置
    ↓
如果同一位置，比较两个提案的预期效果，取更优的

比较标准（按优先级排列）：

| 优先级 | 比较维度 | 更优的定义 |
|--------|---------|-----------|
| 1 | **失败减少幅度** | 预期能减少更多同类失败的提案更优 |
| 2 | **影响范围** | 只影响当前领域的提案优于跨领域影响的提案 |
| 3 | **规则复杂度** | 改动更小、表述更简洁的提案更优（一行规则优于三行） |
| 4 | **可撤销性** | 可以方便回滚的提案优于不可逆的提案 |

如果以上四个维度都分不出优劣 → 保留两个提案，按优先级顺序先后应用（先试 A，验证 3 次不行再换 B）。
```

### 提案与应用冲突

```
已应用一个提案（待验证中）
    ↓
新的聚合发现同类型问题仍存在：
    ↓
① 验证期未结束 → 等待验证完成
② 验证期已结束但效果不佳 → 触发回滚 + 新方案
③ 验证期已结束效果 OK 但还有同类型其他问题 → 新提案（不是回滚）
```

## 治理文件修改优先级总表

| 场景                                 | 推荐修改位置                      | 自动/审批 |
| ------------------------------------ | --------------------------------- | --------- |
| 某种失败反复出现但无规则禁止         | constraint                        | 自动      |
| 某种失败反复出现但有规则但不够明确   | constraint（收紧）                | 自动      |
| 某种失败反复出现且有规则但执行者没看 | heuristic（增加执行流程指引）     | 自动      |
| 开发流程混乱、步骤遗漏               | heuristic（细化步骤）             | 自动      |
| AI 在多个方案间无法选择              | policy（增加决策树）              | 审批      |
| 规则之间自相矛盾                     | policy                            | 审批      |
| 某领域通过率 100% 但规则过多         | constraint/heuristic（放宽/简化） | 审批      |
| 安全遗漏                             | constraint                        | 审批      |
| 新增治理模块/文件                    | 新文件                            | 审批      |
| 删除废弃规则                         | 对应文件                          | 审批      |
