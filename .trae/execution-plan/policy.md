# Policy — 规划阶段决策策略

## 信息不足时

| 情形           | 策略                                  |
| -------------- | ------------------------------------- |
| 需求描述模糊   | 列出待确认项，不假设用户意图          |
| 技术方案不明确 | 提供 2-3 个选项并推荐一个，等用户确认 |
| 项目代码未读过 | 先 Read / Glob 再决策，不凭记忆推测   |

## 风险发现时

| 情形             | 策略                               |
| ---------------- | ---------------------------------- |
| 涉及删除文件     | 计划中高亮标注，执行前确认         |
| 涉及修改公共接口 | 标注影响范围（哪些调用方会受影响） |
| 涉及破坏性变更   | 标注回退路径（如 git 回滚方案）    |

## 依赖冲突时

| 情形         | 策略                     |
| ------------ | ------------------------ |
| 步骤间有依赖 | 依赖在先，被依赖在后     |
| 跨任务有依赖 | 暂停并上报路由协调       |
| 优先级矛盾   | 参考路由冲突优先级表裁决 |

## 跨任务类型切换

同一领域内，不同任务类型可能对应不同 Subagent（如 frontend/create → ui-designer，frontend/modify → frontend-architect）。

| 场景                              | 策略                                                           |
| --------------------------------- | -------------------------------------------------------------- |
| 用户需求明确为单一 task-type      | 按 Agent 映射使用对应 Subagent                                 |
| 用户需求模糊，可跨 task-type      | 优先路由到最匹配的类型，如需切换则由该 Subagent 判断并上报路由 |
| Subagent 执行中发现应切换类型     | 暂停当前任务，记录上下文，上报路由重新分发                     |
| 同一任务需分阶段使用不同 Subagent | 拆分为子任务，在路由层建立依赖链                               |

## 模式选择时

| 情形         | 策略                                |
| ------------ | ----------------------------------- |
| 多个实现方式 | 选择与项目现有风格最接近的          |
| 新的技术选型 | 先 WebSearch 确认最佳实践，再做决定 |
| 代码复用争议 | DRY 优先，但不超过 3 处重复才抽离   |
