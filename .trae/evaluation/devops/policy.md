# Policy — DevOps 评估阶段决策策略

## 通过/不通过决策

| 评估结果                     | 结论           | 后续动作                         |
| ---------------------------- | -------------- | -------------------------------- |
| 语法正确 + 构建通过 + 无 CVE | **通过**       | 标记完成，输出报告               |
| 语法正确 + lint warning      | **有条件通过** | 记录问题，标记完成               |
| 语法错误                     | **不通过**     | 触发 re-execute                  |
| 构建失败                     | **不通过**     | 触发 re-execute                  |
| high/critical CVE            | **不通过**     | 触发 re-plan（更换版本或替代包） |

## 阻断性 vs 非阻断性判定

| 发现            | 分类     | 理由            |
| --------------- | -------- | --------------- |
| YAML 语法错误   | 阻断性   | CI 流程不可执行 |
| Docker 构建失败 | 阻断性   | 部署阻塞        |
| high CVE        | 阻断性   | 安全风险        |
| moderate CVE    | 非阻断性 | 记录移交        |
| 未硬编码密钥    | 阻断性   | 安全问题        |
| 路径引用错误    | 阻断性   | 运行时崩溃      |

## 依赖审计阈值

- critical CVE → 阻断，必须升级
- high CVE → 阻断，评估是否有实际影响
- moderate CVE → 记录，任务完成后统一处理
- low CVE → 记录即可

## 部署回滚决策

- 部署后 85% 以上 endpoint 正常工作 → 非阻断性
- 部署后核心功能不可用 → 阻断性并触发回滚
- 回滚后安排 re-plan 修复后再部署
