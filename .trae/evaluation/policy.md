# Policy — 评估阶段通用决策策略

触发 re-execute 或 re-plan 时，循环治理规则见 `evaluation/loop-governance.md`。

## 通过/不通过决策

| 评估结果                      | 结论           | 后续动作                                  |
| ----------------------------- | -------------- | ----------------------------------------- |
| 所有门禁通过 + 完整性校验通过 | **通过**       | 标记任务完成，输出评估报告                |
| 门禁通过 + 非阻断性问题       | **有条件通过** | 记录问题清单，标记任务完成                |
| 门禁失败                      | **不通过**     | 附完整错误信息，触发 re-execute（≤3 次）  |
| 计划文件缺失或变更不一致      | **不通过**     | 附缺失/差异清单，触发 re-execute（≤3 次） |
| 回归发现阻断性 Bug            | **不通过**     | 附回归发现，触发 re-plan（≤2 次）         |
| 依赖链上下文丢失或不完整      | **不通过**     | 附缺失信息，触发上一步骤恢复              |

## 阻断性 > 非阻断性判定

- 类型/语法错误 → 阻断性
- lint error → 阻断性
- lint warning → 非阻断性
- 计划文件缺失 → 阻断性
- 计划外文件被修改 → 阻断性
- 性能退化 10% 内 → 非阻断性；以上 → 阻断性并上报
- 安全隐患 → 阻断性并上报路由

## 评估证据优先级

| 证据类型                  | 可信度 |
| ------------------------- | ------ |
| 命令输出（stdout/stderr） | 高     |
| 文件内容（Read）          | 高     |
| git diff                  | 高     |
| grep/glob 搜索            | 中     |
| AI 自主判断               | 低     |

## 部分通过场景

- 核心功能完整度 ≥ 80% → 有条件通过，附待修复清单
- 核心功能完整度 < 80% → 不通过，触发 re-plan
- **完整度计算方式**：`完整度 = 已通过检查项数 / 完成检查清单总项数 × 100%`
- 核心功能列表取自对应 workflow 的完成检查清单

## 复现失败处理

- 第 1 次 → 修复后重新评估
- 第 2 次 → 检查修复逻辑是否正确
- 第 3 次 → 暂停评估，上报路由协调人工介入
