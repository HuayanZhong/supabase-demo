# Heuristic — 质量评估阶段最佳实践

## 评估前加载顺序

```
1. workflows/quality/{task-type}.md 完成检查清单  → 评估基准
2. execution-plan/quality/（3 文件）              → 规划承诺
3. execution-engine/quality/（3 文件）            → 执行记录
```

## 评估流程

### 测试（test）

```
① 全量测试运行        → pnpm test 通过
② 新增测试确认        → 计划中添加的测试文件存在
③ 回归测试确认        → 已有测试未因本次变更而新增失败
④ 测试覆盖率检查      → 新增代码有对应测试
⑤ 多余文件检查        → 无未列入计划的测试文件
```

### 代码审查（review）

```
① 审查输出完整性      → 有明确的审查结果（通过/有条件/不通过）
② 问题分类            → 阻断性/非阻断性归类
③ 问题追溯            → 每个问题标注文件路径和行号
④ 修复确认            → 阻塞性问题修复后重新审查
⑤ 多余文件检查        → 无未列入计划的文件
```

### 性能评估（perf）

```
① 基线数据确认        → 有变更前的性能基线数据
② 当前数据采集        → 按基线相同的条件采集数据
③ 对比分析            → P50/P95/P99 逐项对比
④ 退化判定            → 超过 10% 的设备来源分析
⑤ 优化建议            → 如有退化，给出可操作的优化方向
```

### 安全审计（security）

```
① 审计范围确认        → 覆盖计划中的所有模块
② 问题记录            → 每项问题标注 CWE/OWASP 分类
③ 严重等级判定        → high/moderate/low 三级
④ 修复跟踪            → high 标注已修复 / 待修复
⑤ 余下风险说明        → medium/low 问题排期说明
```

### API 测试（api-test）

```
① endpoint 覆盖       → 计划中所有 endpoint 已测试
② 状态码验证          → 预期状态码（200/201/400/401/404 等）
③ 响应体验证          → 预期响应结构
④ 边界测试            → 空参数/类型错误/超长输入
⑤ 认证测试            → 未认证请求是否正确返回 401
```

## 全局优先级

质量评估先于业务功能评估执行：

```
security audit  >  perf  >  test  >  api-test  >  review
```

安全 > 性能 > 测试 > API 测试 > 审查
