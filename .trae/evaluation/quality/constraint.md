# Constraint — 质量评估阶段硬约束

## 评估前约束

- 开始评估前，必须先读取 `execution-engine/quality/` 了解执行约束
- 开始评估前，必须先读取 `workflows/quality/{task-type}.md` 的完成检查清单
- 开始评估前，必须先读取 `execution-plan/quality/` 确认规划承诺

## 验证门禁

1. **测试覆盖** — 全量测试运行通过（`pnpm test`）
2. **安全审计** — 无新增 high/critical CVE
3. **性能基线** — 无可量化的性能退化（P99 响应时间退化 < 10%）
4. **类型检查** — `pnpm check-types` 零错误
5. **API 测试** — 被测 endpoint 返回预期响应

## 完整性检查

- 审查任务（review）必须输出结构化审查结果：问题清单、严重等级、修复建议
- 性能评估（perf）必须输出量化指标：P50/P95/P99、请求耗时分布
- 安全审计（security）必须输出问题清单，逐项标注已修复/待修复/误报
- API 测试（api-test）必须输出测试结果表格：endpoint → 状态码 → 响应体

## 范围约束

- 发现的问题按严重等级分级处理，阻断性问题必须先修复
- 性能退化超过 10% 必须上报路由
- 安全发现不可忽——高危必须立即修复，中危必须记录并排期

## 输出约束

- 所有评估必须输出结构化报告
- 测试评估报告包含：测试项总数、通过数、失败数、失败详情
- 审查/安全评估报告包含：问题清单（严重等级、优先级、状态）
- 性能评估报告包含：基线数据 vs 当前数据对比表
- API 测试报告包含：endpoint 完整性与正确性
