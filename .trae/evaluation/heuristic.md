# Heuristic — 评估阶段通用最佳实践

## 评估前加载顺序

```
1. workflows/{domain}/{task-type}.md 完成检查清单  → 明确评估基准
2. execution-plan/{domain}/（3 文件）              → 明确规划承诺
3. execution-engine/{domain}/（3 文件）            → 明确执行记录
4. 读取实际产出文件                                → 对比预期
```

## 通用评估流程

```
① 读取完成检查清单    → 明确本次任务的目标和验收标准
② 验证文件完整性      → 计划文件是否存在/是否被修改
③ 执行质量门禁        → 类型/lint/format/build
④ 回归验证            → 引用方是否正常工作
⑤ 范围检查            → 无计划外文件、无越界修改
⑥ 输出评估报告        → 结构化、可追溯
⑦ 写入经验数据       → 按 evolution/heuristic.md 格式写入 .trae/experience/
```

## 证据采集通用方法

- 类型/lint/format 结果 → 保存命令输出片段
- 文件存在性 → 用 `Glob` 或 `LS` 确认
- 文件变更 → 用 `git diff`（如适用）确认修改范围
- 依赖链上下文 → 读取上一步输出的 context 确认传递完整性

## 回归范围建议

| 变更类型      | 回归方式                          |
| ------------- | --------------------------------- |
| 修改公共模块  | 搜索该模块的所有引用              |
| 修改 API 结构 | 搜索调用方并确认请求/响应结构一致 |
| 修改类型定义  | 搜索该类型的所有引用方            |
| 修改导入路径  | 确认所有引用方已更新              |
| 删除文件/模块 | 确认无残留引用                    |

## 多任务评估策略

- 依赖链任务：按步骤顺序逐项评估，前一步不通过则暂停后续
- 并行子任务：各自独立评估，汇总时合并报告
- 执行中期发现新依赖：新任务完成后追加评估，再继续原任务
