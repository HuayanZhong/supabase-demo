# Constraint — 前端评估阶段硬约束

## 评估前约束

- 开始评估前，必须先读取对应 `execution-engine/frontend/` 了解执行阶段的约束/启发/策略
- 开始评估前，必须先读取 `workflows/frontend/{task-type}.md` 的完成检查清单，作为评估基准
- 开始评估前，必须先读取 `execution-plan/frontend/` 确认规划期的承诺与约束
- 评估对象必须是执行阶段的实际产出文件，不得基于记忆或预期

## 验证门禁

以下门禁**必须全部通过**才能标记为完成：

1. **类型检查** — `pnpm check-types` 零错误
2. **代码质量** — `pnpm lint` 零 error（warning 可容忍但需记录）
3. **格式化** — `pnpm format` 无未格式化文件
4. **编译验证** — `pnpm build`（如适用）编译通过

任一门禁未通过，不得输出来完成结论。

## 完整性检查

- 计划中列出的新建文件必须真实存在，逐一确认文件路径
- 计划中列出的修改文件必须真实被修改，逐一确认变更点
- 计划中列出的删除文件必须已删除或标记废弃
- 计划中标注的"只读"文件不得有变更
- 翻译 key 清单中标注"转 shared"的，必须确认已记录移交

## 范围约束

- 评估仅对当前任务的产出负责，不评估无关代码
- 发现任务范围外的 Bug 时，记录但不阻断，标注"非本次任务范围"
- 发现任务范围外的安全隐患时，记录并上报路由
- 评估结论必须基于可复现的证据（命令输出、文件内容），不得凭感觉判断

## 回归约束

- 涉及修改已有代码的任务，必须执行回归验证
- 回归范围至少覆盖：被修改文件的相关引用方、被修改组件的调用方
- 回归发现的问题按阻断性/非阻断性分类记录

## 输出约束

- 评估结论必须输出结构化报告，包含：评估项清单、每项通过/失败状态、失败原因
- 评估报告必须是可复现的（他人可阅读并理解）
- 结论有两种：**通过**（标记任务完成）/ **不通过**（附原因并触发 re-plan 或 re-execute）
