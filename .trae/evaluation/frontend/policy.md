# Policy — 前端评估阶段决策策略

## 通过/不通过决策

| 评估结果                                            | 结论           | 后续动作                         |
| --------------------------------------------------- | -------------- | -------------------------------- |
| 所有门禁通过 + 完整性校验通过                       | **通过**       | 标记任务完成，输出评估报告       |
| 门禁通过 + 非阻断性问题（lint warning、未使用变量） | **有条件通过** | 记录问题清单，标记任务完成       |
| 门禁失败（check-types error、lint error）           | **不通过**     | 附完整错误信息，触发 re-execute  |
| 计划文件缺失或变更不一致                            | **不通过**     | 附缺失/差异清单，触发 re-execute |
| 回归发现阻断性 Bug                                  | **不通过**     | 附回归发现，触发 re-plan         |

## 阻断性 vs 非阻断性判定

| 发现                      | 分类       | 理由                         |
| ------------------------- | ---------- | ---------------------------- |
| pnpm check-types 有 error | 阻断性     | 类型安全是前提               |
| pnpm lint 有 error        | 阻断性     | 代码质量门禁                 |
| pnpm lint 有 warning      | 非阻断性   | 可容忍，记录即可             |
| 计划文件缺失              | 阻断性     | 产出不完整                   |
| 计划外文件被修改          | 阻断性     | 范围溢出                     |
| 语义色被误用为硬编码      | 非阻断性   | 可后续修复                   |
| 翻译 key 缺失             | 非阻断性   | 记录移交即可                 |
| 性能退化                  | 视严重程度 | 10% 内非阻断，以上阻断并上报 |

## 评估证据优先级

| 证据类型       | 可信度 | 说明                             |
| -------------- | ------ | -------------------------------- |
| 命令输出       | 高     | RunCommand 的 stdout/stderr      |
| 文件内容       | 高     | Read 获取的当前文件内容          |
| grep/glob 结果 | 中     | 搜索到的引用/文件列表            |
| AI 自主判断    | 低     | 凭经验而非数据的结论（尽量避免） |

评估结论应优先依赖高可信度证据。

## 多任务评估顺序

| 场景               | 策略                                                    |
| ------------------ | ------------------------------------------------------- |
| 单领域单任务       | 按单评估流程执行，无特殊处理                            |
| 依赖链多步骤       | 每步完成后立即评估，评估通过才进入下一步                |
| 并行子任务         | 各自独立评估，汇总结果时合并报告                        |
| 执行中期发现新依赖 | re-route 追加子任务 → 新任务完成后评估 → 继续原任务评估 |

## 复现失败的处理

| 失败次数 | 处理方式                       |
| -------- | ------------------------------ |
| 第 1 次  | 修复后重新评估                 |
| 第 2 次  | 评估者自行检查修复逻辑是否正确 |
| 第 3 次  | 暂停评估，上报路由协调人工介入 |

同一个评估项连续 3 次失败说明不是简单的问题，需要升级处理。

## 部分通过的场景

当任务产出部分符合预期但部分有偏差时：

- 核心功能完整度 ≥ 80% → 有条件通过 + 记录待修复项
- 核心功能完整度 < 80% → 不通过，触发 re-plan
- 核心功能列表取自对应 workflow 的完成检查清单
