# Loop Governance — 评估循环治理

## 概述

当 Evaluation 输出 **不通过** 时，进入循环治理。循环治理控制 Full Loop Engineering 的收敛过程，确保：

1. 循环有限步内结束（不陷入死循环）
2. 每轮循环都有有效进展
3. 达到上限后自动升级处理

```
Evaluation → 不通过
    ↓
┌─ re-execute ──────────────────────────────┐
│  同层重试：修复具体问题，不改变设计方案     │
│  适用：check-types error、lint error、文件  │
│  遗漏等可量化问题                           │
└────────────────────────────────────────────┘
    ↓ 连续重试无效
┌─ re-plan ──────────────────────────────────┐
│  升级重来：修改设计方案、调整约束/策略      │
│  适用：架构缺陷、设计矛盾、重复性问题        │
└────────────────────────────────────────────┘
    ↓ 规划无效
┌─ 人工介入 ──────────────────────────────────┐
│  上报路由协调人工处理                        │
│  适用：3 次循环无进展、安全漏洞无法修复      │
└─────────────────────────────────────────────┘
```

---

## 循环进入日志

**每次进入循环治理时，必须输出以下日志**（日志格式标准见 `../logging.md`）：

```
[LOOP:enter]      START  | 进入循环治理           | task_id=任务ID;trigger=评估不通过;conclusion=结论
[LOOP:cycle]      RETRY  | re-execute 第N次        | attempts=N/3;failure=失败类型
[LOOP:lock]       LOCK   | 锁定检查项             | check=检查项;cycle=N;locked_count=N
[LOOP:lock]       UNLOCK | 解锁检查项             | check=检查项;trigger=解锁原因
[LOOP:lock]       CLEAR  | 清空锁定               | trigger=清空原因;locked_count=N
[LOOP:lock]       EXPIRE | 锁定过期               | check=检查项;cycle=N;action=自动解锁
[LOOP:retry-wait] OK     | 退避等待               | wait_ms=等待毫秒数
[LOOP:tool-dedup] SKIP   | 工具去重               | action=操作;hash=哈希;reason=重复
[LOOP:semantic]   BLOCKED| 语义循环检测            | pattern=模式;conclusion=结论
[LOOP:convergence]STALL  | 收敛停滞               | feature_hash=哈希;match=2/2;action=升级re-plan
[LOOP:convergence]DEADLOCK| 收敛死锁              | feature_hash=哈希;failures_same=true;action=人工介入
[LOOP:convergence]OK     | 有效收敛               | feature_hash=哈希;failure_reduction=百分比
[LOOP:returns]    WARN   | 收益递减               | delta=百分比;threshold=阈值;consecutive=N;action=强制升级
[LOOP:returns]    FAIL   | 收益退步               | delta=百分比;action=立即熔断
[LOOP:returns]    OK     | 有效改进               | delta=百分比;threshold=阈值;status=继续
[LOOP:cost]       WARN   | 调用数预警             | calls=N/30
[LOOP:anchor]     OK/FAIL| 目标锚定              | deviation=偏差百分比
[LOOP:exit]       END    | 退出循环               | exit_by=原因;cycles=N;result=结果
```

LOOP:start 之后每一步都有记录，任何步骤 FAIL 或 BLOCKED 都会影响循环走向。以上日志是循环治理的必输项，不得跳过。

## Constraint（硬约束）

### 循环计数

- 同一任务的 **re-execute 最大次数**：3 次
- 同一任务的 **re-plan 最大次数**：2 次
- 同一个 Evaluation 任务从首次执行到最终通过的 **总循环上限**：5 次（re-execute + re-plan 合计）
- 达到上限仍未通过 → 强制升级到人工介入

### 循环范围

- 每次循环均限定在当前任务范围内，不得跨越到新任务
- 依赖链任务中某一步失败，只循环该步骤，不重跑已通过的步骤
- 并行子任务中某一条失败，只循环失败子任务，不影响其他已完成子任务

### 循环变化检测

- re-execute 第 2 次时，必须对比第 1 次的失败原因。如果**完全相同**，禁止第 2 次 re-execute，强制升级到 re-plan
- re-plan 第 2 次时，必须对比第 1 次的方案。如果方案没有实质性改变（仅措辞调整），禁止第 2 次 re-plan，强制升级到人工介入

### 每次循环的最低产出

- 每次 re-execute 后的产出必须与前一次**有可量化的改进**（减少的 lint error 数、修复的文件数等）
- 每次 re-plan 后的方案必须与前一次**有实质性差异**（不同的架构方式、不同的工具选择）
- 无实质改进的循环视为"无效循环"，直接升级

### 状态锁定（State Locking）

防止循环中已通过的检查项反复被重试，浪费工具调用和上下文空间。

**锁定机制：**

每次 re-execute 完成后，将本轮通过的所有检查项加入"已通过列表"：

```json
{
  "locked_checks": [
    { "check": "pnpm check-types", "passed_since": "cycle-1", "status": "locked" },
    { "check": "pnpm lint", "passed_since": "cycle-2", "status": "locked" }
  ]
}
```

该列表写入循环状态记录的 `locked_checks` 字段。

**锁定规则：**

| 规则         | 说明                                                                     |
| ------------ | ------------------------------------------------------------------------ |
| 锁定范围     | 仅限于当前任务内的非安全类检查                                           |
| 锁定生效域   | re-execute 锁定只在本轮 re-execute 内有效，re-plan 开始后清空            |
| 跳过检查     | 被锁定的检查项不在后续循环的 fail 清单中，不占用重试次数                 |
| 可锁定类型   | lint 检查、check-types、单文件格式验证、模块注册检查                     |
| 不可锁定类型 | 安全检查（始终重检）、集成测试（因其他修改可能间接受损）、跨文件影响验证 |

**解锁条件：**

锁定不是永久性的，以下情况解锁：

| 触发条件                       | 处理方式                             |
| ------------------------------ | ------------------------------------ |
| 另一项修复导致已锁定的检查报错 | 该检查项自动解锁，重新加入失败列表   |
| re-plan 方案变更               | 清空所有锁定                         |
| 人工介入后指定                 | 按人工指示处理                       |
| 超过 3 轮循环                  | 自动解锁（防止因锁定掩盖了退化问题） |

**日志输出：**

```
[LOOP:lock] LOCK  | 锁定检查项           | check=pnpm check-types;cycle=2;action=跳过后续检查
[LOOP:lock] UNLOCK| 解锁检查项           | check=pnpm lint;trigger=因另一修复报错;action=重新加入失败列表
[LOOP:lock] CLEAR  | 清空锁定             | trigger=re-plan;locked_count=3
[LOOP:lock] EXPIRE | 锁定过期             | check=pnpm format;cycle=4;max=3;action=自动解锁
```

### 工具调用去重

- 同一轮循环中，**禁止使用完全相同的参数重复调用同一个工具**
- 每次工具调用后，记录 `工具名 + 参数摘要（取前 100 字符的哈希）` 到循环状态中
- 下次调用前，先检查是否已存在相同记录
- 如果检测到重复调用意图：
  - 第 1 次重复 → 跳过调用，直接复用上次结果
  - 第 2 次重复 → 视为"无效循环"，升级处理
- 依赖链中不同步骤间的同一工具调用不适用此约束（前一步写入 → 下一步读取是正常流程）

### 静默失败检测

- 每次工具调用后，**必须验证返回结果的有效性**
- 以下情况视为静默失败，不得继续向下执行：
  - 工具返回空字符串或空对象（`{}`）
  - 工具返回 `null` 或 `undefined`
  - 工具返回异常状态码但无错误信息
  - 工具返回了预期结构但不包含实际内容（如空数组、空文件内容）
- 检测到静默失败时的处理流程：
  ```
  检测到静默失败
    → 重新调用一次工具（最多 1 次重试）
    → 仍然静默失败？→ 上报异常，触发 re-execute
    → 连续 2 轮静默失败？→ 升级到 re-plan
  ```
- 依赖链上下文中传递空值也按静默失败处理

### 重试退避

- 连续三次或以上的工具调用失败，必须使用指数退避：
  ```
  第 1 次失败 → 立即重试（不退避）
  第 2 次失败 → 等待 1 秒后重试
  第 3 次失败 → 等待 2 秒后重试
  第 4 次失败 → 等待 4 秒后重试
  第 N 次失败 → 等待 2^(N-2) 秒后重试（上限 60 秒）
  ```
- 每次退避等待需加入 ±20% 的随机抖动（jitter），避免"惊群效应"
- 退避期间应标记该工具为"不稳定"状态，本轮循环中优先考虑替代方案
- 如果退避超 60 秒仍失败 → 视为该工具当前不可用，按 MCP/Skill 降级策略处理

### 成本 / 预算约束

- 单次循环的总工具调用次数上限：**30 次**（含所有工具的累计调用）
- 达到 30 次后，无论是否还有可用步数，强制触发 re-plan 或人工介入
- AI 应记录每次工具的消耗情况（调用次数、结果类型），在状态中追踪
- 如果某次循环消耗明显超出预期（如同一工具调用超 5 次），暂停并输出成本预警

### 收敛检测（Convergence Detection）

检测循环是否陷入"原地踏步"——每次都在改，但产出结果一模一样。

> **统一计数器：** 使用单一 `no_progress_count`，不再区分 stall_count 和 deadlock_count。
> 凡"失败列表未减少"或"产出特征完全相同"均 +1，连续 2 次强制升级。
> 这避免了 AI 在两种状态间交替跳变绕过升级。

**特征提取：**

每次 re-execute 完成后，提取本次产出的关键特征：

- 失败文件列表（排序后取 SHA256 哈希的前 12 位）
- 错误类型列表（去重排序后取哈希）
- 修复前后的代码 diff（取变更行数 + 变更文件数）

**对比与判定：**

```
当前特征 vs 上次特征
    ↓
├── 产出特征完全相同 → no_progress_count +1
│   说明 AI 在重复输出相同内容
│
├── 特征不同但失败列表完全相同 → no_progress_count +1
│   说明在同一个问题上反复做不同尝试但都没用
│
├── 特征不同且失败列表减少 → 检查改进幅度
│   │
│   ├── 改进幅度 ≥ 10%（失败减少 ≥ 10%）→ no_progress_count 归零，正常继续
│   │
│   └── 改进幅度 < 10%（改进不足）→ no_progress_count +1
│       收益递减，连续 2 次触发熔断
│
└── 失败列表增加（退步）→ 立即熔断，不等连续次数

no_progress_count ≥ 2 → 强制升级
    re-execute 场景 → 升级到 re-plan
    re-plan 场景 → 升级到人工介入
```

**日志输出：**

```
[LOOP:convergence] STALL    | 无进展              | reason=特征相同/no_progress=N;action=升级re-plan
[LOOP:convergence] DEADLOCK | 无进展              | reason=失败列表相同/no_progress=N;action=升级人工
[LOOP:convergence] DIMINISH | 收益递减            | delta=改进幅度;no_progress=N;action=升级
[LOOP:convergence] REGRESS  | 收益退步            | delta=+N%;action=立即熔断
[LOOP:convergence] OK       | 有效收敛            | feature_hash=xxx;failure_reduction=N%;no_progress=0
```

收敛检测结果写入循环状态记录的 `convergence` 字段。

### 收益递减检测（Diminishing Returns）

收益递减的判定已合并到上方收敛检测的 `no_progress_count` 统一计数器中。

**改进幅度定义：**

| 循环类型   | 改进幅度计算                             | 有效阈值                  |
| ---------- | ---------------------------------------- | ------------------------- |
| re-execute | `(本次失败数 - 上次失败数) / 上次失败数` | ≤ -10%（失败减少 ≥ 10%）  |
| re-plan    | 方案关键决策点差异比例                   | ≥ 30%（方案有实质性变更） |

**判定规则（与收敛检测统一）：**

| 改进幅度 delta（负=改进，正=退步） | 判定     | no_progress_count |
| ---------------------------------- | -------- | ----------------- |
| delta > 0（失败增加）              | 收益退步 | 立即熔断          |
| -10% < delta ≤ 0（改进不足 10%）   | 收益递减 | +1，连续 2 次熔断 |
| delta ≤ -10%（失败减少 ≥ 10%）     | 有效改进 | 归零              |

**首次迭代处理：**

- 首次 re-execute 没有"上次对比"，跳过收益递减检测
- 从第 2 次 re-execute 开始启用
- 首次 re-plan 同样跳过，从第 2 次 re-plan 开始启用

收益递减检测结果写入循环状态记录的 `diminishing_returns` 字段。

## Heuristic（启发式）

### 循环执行流程

```
Evaluation 不通过
    ↓
检查失败类型：

check-types error / lint error / 文件遗漏
    → re-execute（同层重试）
    → 修复 → 重新评估
    → 通过？→ 结束
    → 不通过？→ 检查失败原因与上次是否相同
        → 不同？→ 再试（计数 +1）
        → 相同？→ 升级到 re-plan

架构缺陷 / 设计矛盾 / 重复问题
    → re-plan（升级重来）
    → 修改方案 → 重新路由 → 重新执行 → 重新评估
    → 通过？→ 结束
    → 不通过？→ 检查方案是否实质性改变
        → 有改变？→ 再试（计数 +1）
        → 无改变？→ 升级到人工介入

安全隐患 / 未知错误 / 3 次无效
    → 直接人工介入
    → 上报路由协调
```

### 循环状态记录

每轮循环必须记录以下状态，供对比和决策：

```json
{
  "cycle": 1,
  "type": "re-execute",
  "failed_items": [
    {
      "check": "pnpm check-types",
      "error_count": 3,
      "error_location": "src/components/Form.vue:45"
    }
  ],
  "locked_checks": [{ "check": "pnpm lint", "passed_since": "cycle-1", "status": "locked" }],
  "previous_failure_overlap": false,
  "improvement_metric": "-50% error",
  "convergence": {
    "status": "effective",
    "feature_hash": "a1b2c3d4e5f6",
    "previous_feature_hash": "d4e5f6a1b2c3",
    "no_progress_count": 0,
    "last_reason": null
  },
  "diminishing_returns": {
    "delta": -0.5,
    "threshold": -0.1,
    "below_threshold": false,
    "consecutive_below": 0,
    "regression": false
  }
}
```

**每轮循环迭代完成后，该状态记录必须同步写入 `.trae/memory/experience/`**，供 evolution 收集。文件名命名规则：

```
.trae/memory/experience/{domain}/loop-{task-id}-cycle-{n}-{date}.json
```

最终评估通过后，再提交 1 条聚合后的完整经验记录（覆盖首次执行 + 所有循环迭代）。evolution 聚合时会按 `task_id` 将循环记录和最终记录合并。

### re-execute → re-plan 切换判断

使用以下判断表决定是否切换：

| 当前状态                                     | 建议动作               |
| -------------------------------------------- | ---------------------- |
| 第 1 次 re-execute 失败                      | 允许第 2 次 re-execute |
| 第 2 次 re-execute 失败（新问题）            | 允许第 3 次 re-execute |
| 第 2 次 re-execute 失败（与第 1 次相同问题） | **切换到 re-plan**     |
| 第 3 次 re-execute 失败                      | **切换到 re-plan**     |
| 第 1 次 re-plan 失败（新方案）               | 允许第 2 次 re-plan    |
| 第 1 次 re-plan 失败（方案无实质改变）       | **切换到人工介入**     |
| 第 2 次 re-plan 失败                         | **切换到人工介入**     |

### 无效循环检测方式

- 对比失败项目列表：用 `grep` 或字符串匹配检查前次失败的文件路径和错误信息
- 对比评估报告：读取前次评估报告的输出，逐项对比
- 如果完全一致 → 判定为无效循环
- 如果部分一致但上次修复未生效 → 判定为新问题（有效循环）

### 语义循环检测

硬性的"完全相同"检测不够，需要识别"换了说法但本质相同"的语义循环：

| 语义循环模式                                                       | 检测方式                                       | 判定     |
| ------------------------------------------------------------------ | ---------------------------------------------- | -------- |
| 同一文件同一行报同一类错误                                         | 检查失败文件路径 + 行号范围是否重叠            | 语义循环 |
| 相同工具名 + 参数结构相似（如换了一个无关参数）                    | 检查工具名 + 参数 key 集合是否一致             | 语义循环 |
| 修复方案描述不同但实际做了同一件事                                 | 对比前后两次 Read 同一文件，检查 diff 是否一致 | 语义循环 |
| 连续 2 次失败报错信息包含相同的模板（如 `Cannot find module '*'`） | 用正则提取错误模板，去变量后对比               | 语义循环 |
| 重新打开已关闭的问题                                               | 检查文件路径在上次失败中出现过                 | 语义循环 |

检测到语义循环时的处理方式：

- 第 1 次语义循环 → 记录"疑似语义循环"，允许一次差异化尝试（换完全不同的方案）
- 第 2 次语义循环 → 判定为无效循环，强制升级到 re-plan（不等待次数上限）
- 语义循环的检测优先级高于普通失败对比——即如果同时有硬性不同但语义相同，按语义循环处理

### 上下文压缩策略

长循环中上下文会持续膨胀，导致模型降智和成本飙升。必须在循环过程中主动压缩：

```
每 2 次 re-execute 或每 5 次工具调用后（以先到为准）：
    → 读取循环状态记录中的 summary 字段
    → 将历史工具调用详情压缩为"已做的尝试及其结果"摘要
    → 丢弃历史中不再需要的原始输出（保留摘要版本）
    → 保留以下不可压缩的信息：
       - 原始任务目标
       - 已完成的步骤清单
       - 当前失败的类型和位置
```

可压缩的内容：

| 可以压缩                     | 不可压缩           |
| ---------------------------- | ------------------ |
| 各工具调用的完整返回体       | 原始任务目标       |
| 中间推理步骤的完整思维链     | 失败错误信息原文   |
| 依赖链中上一步的完整 context | 已完成的步骤清单   |
| 评估报告的详细输出           | 人工介入的历史记录 |

压缩后保留 3 个关键段：

```
[目标] 原始任务目标（保持不变）
[进度] 当前已完成的步骤清单 + 当前循环次数
[失败] 本轮失败的检查项 + 错误位置 + 已尝试的修复方案摘要
```

### 目标锚定机制

每次进入新循环迭代前，必须先做目标一致性检查：

```
① 读取原始任务目标（来自 workflow 或 execution-plan）
② 对比当前正在执行的内容
③ 偏差度评估：
   - 0% → 完全一致，继续
   - < 20% → 有偏差但可接受，修正后继续
   - ≥ 20% → 已脱离原始目标，暂停并上报
```

偏差度评估参考：

| 表现                                | 偏差度 |
| ----------------------------------- | ------ |
| 仍在做原始任务，只是换了实现方式    | 0%     |
| 做了原始任务 + 额外做了附近的小优化 | < 20%  |
| 原始任务没做完就去解决依赖项的问题  | ≥ 20%  |
| 原始任务被搁置，转向了另一个方向    | ≥ 50%  |

检测到目标漂移后：

- 偏差度 < 20% → 放弃额外工作，回到原始任务
- 偏差度 ≥ 20% → 暂停当前工作，输出"目标漂移报告"到路径，触发 re-plan
- 偏差度 ≥ 50% → 直接上报人工介入

### 状态持久化方案

循环状态必须在每一步持久化到可复现的路径，确保中断后可恢复：

```
每完成一次 re-execute 或 re-plan 后：
    → 写入 .trae/loop-state/{task-id}-cycle-{n}.json
    → 文件内容包括循环状态 JSON（循环次数、类型、失败项列表）
    → 保留所有历史状态文件，不覆盖（方便追溯）
```

状态文件命名规则：

```
.trae/loop-state/{task-hash}-cycle-001.json
.trae/loop-state/{task-hash}-cycle-002.json
```

如果循环被中断（进程重启、超时等）：

```
① 扫描 .trae/loop-state/ 下是否存在该 task 的最近状态文件
② 读取最新循环次数 + 失败项
③ 从该次数继续，不从头开始
④ 如果状态文件损坏或格式不正确 → 从头开始（安全降级）
```

## Policy（决策策略）

### 循环次数上限汇总

| 指标                                | 上限 | 超出后         |
| ----------------------------------- | ---- | -------------- |
| re-execute 次数                     | 3    | 强制 re-plan   |
| re-plan 次数                        | 2    | 强制人工介入   |
| 总循环数（含 re-execute + re-plan） | 5    | 强制人工介入   |
| 无效循环次数                        | 1    | 强制升级下一级 |
| 收敛停滞次数                        | 1    | 强制 re-plan   |
| 收敛死锁次数                        | 1    | 强制人工介入   |
| 收益递减连续次数                    | 2    | 强制升级下一级 |
| 收益退步次数                        | 1    | 立即熔断       |

### 失败分类 → 动作映射

| 失败类型                 | 首轮动作                   | 后续升级路径                          |
| ------------------------ | -------------------------- | ------------------------------------- |
| check-types error        | re-execute                 | re-execute → re-plan → 人工           |
| lint error               | re-execute                 | re-execute → re-plan → 人工           |
| 计划文件缺失             | re-execute                 | re-execute → re-plan                  |
| 计划外文件               | re-execute                 | re-execute                            |
| Migration 无 down        | re-execute                 | re-execute                            |
| 模块注册遗漏             | re-execute                 | re-execute                            |
| Breaking change 未标注   | re-execute                 | re-execute                            |
| API key 硬编码           | **人工介入**（直接）       | 安全事件                              |
| 性能退化 > 10%           | re-plan                    | re-plan → 人工                        |
| 安全审计 high 未修复     | **人工介入**（直接）       | 安全事件                              |
| 依赖链 context 丢失      | re-execute（重跑该步骤）   | re-execute → re-plan                  |
| 语义错误                 | re-plan                    | re-plan → 人工                        |
| 工具调用重复             | 跳过调用，复用上次结果     | 第 2 次重复 → 无效循环升级            |
| 静默失败（工具返回空）   | 重新调用 1 次              | 连续 2 轮 → re-plan                   |
| 退避超 60 秒             | 标记工具不可用，走替代方案 | 替代方案失败 → re-plan                |
| 调用次数超 30            | 成本熔断 → 强制 re-plan    | 人工介入                              |
| 语义循环                 | 允许一次差异化尝试         | 第 2 次 → 直接 re-plan（不等上限）    |
| 收敛停滞                 | 连续 2 次 → 强制 re-plan   | re-plan → 人工                        |
| 收敛死锁                 | 连续 2 次 → 强制人工介入   | —                                     |
| 收益递减                 | 连续 2 次 → 强制升级下一级 | re-execute → re-plan / re-plan → 人工 |
| 收益退步                 | 立即熔断                   | 强制升级                              |
| 目标漂移 ≥ 20%           | 暂停，触发 re-plan         | ≥ 50% → 人工介入                      |
| 循环中断（状态丢失）     | 从状态文件恢复             | 无状态文件 → 从头开始                 |
| 锁定检查项被其他修复破坏 | 自动解锁，重新加入失败列表 | 连续 2 次解锁 → 该检查项不可锁定      |
| 锁定过期（超 3 轮）      | 自动解锁，重新检查         | —                                     |

### 人工介入的触发条件

以下任一条件满足，强制触发人工介入：

1. re-execute 达到 3 次上限仍未通过
2. re-plan 达到 2 次上限仍未通过
3. 总循环数达到 5 次上限
4. 检测到无效循环（连续两次相同失败）
5. 安全相关失败（硬编码密钥、SQL 注入等）
6. 评估发现无法在当前治理框架内解决的问题

### 人工介入后的流程

```
人工介入触发
    ↓
输出完整报告：
  - 当前循环次数和类型
  - 失败项目清单（含历史对比）
  - 已尝试的修复方案
  - 建议下一步方向
    ↓
上报到路由 → 等待人工决策
    ↓
人工决策结果：
  ① 认可方案 → 手动批准 → 继续执行
  ② 否定方案 → 提供新方向 → 重新路由
  ③ 取消任务 → 标记为不可行
```

### 循环异常降级总表

| 异常场景                       | 处理方式                     |
| ------------------------------ | ---------------------------- |
| 同问题连续重试 3 次            | 升级 re-plan                 |
| re-plan 出相同方案             | 升级人工介入                 |
| 安全漏洞循环中未修             | 立即升级人工介入             |
| 总循环超 5 次                  | 强制终止，人工介入           |
| 依赖链中间步骤成功但下游重新跑 | 不允许，保持中间步骤缓存     |
| 并行子任务一失败拖慢其他       | 独立循环，不阻塞并行任务     |
| AI 自主在循环中扩大任务范围    | 阻断并回归原任务范围         |
| 工具调用重复（第 2 次）        | 判定无效循环，升级 re-plan   |
| 静默失败连续 2 轮              | 升级 re-plan                 |
| 退避超 60 秒无替代方案         | 升级 re-plan                 |
| 工具调用次数超 30              | 成本熔断，强制 re-plan       |
| 语义循环第 2 次                | 直接 re-plan（跳过次数上限） |
| 收敛停滞第 2 次                | 强制 re-plan                 |
| 收敛死锁第 2 次                | 强制人工介入                 |
| 收益递减连续 2 次              | 强制升级下一级               |
| 收益退步（失败数增加）         | 立即熔断，强制升级           |
| 目标漂移 ≥ 20%                 | 暂停，触发 re-plan           |
| 目标漂移 ≥ 50%                 | 直接人工介入                 |
| 循环中断且有状态文件           | 从最近状态文件恢复           |
| 循环中断且无状态文件           | 从头开始（安全降级）         |
| 锁定检查项因另一修复报错       | 自动解锁，重新加入失败列表   |
| 同一检查项重复解锁 ≥ 2 次      | 标记该检查项不可锁定         |
| 锁定过期（超过 3 轮循环）      | 自动解锁，重新执行检查       |
