# Loop Governance — 评估循环治理

## 概述

当 Evaluation 输出 **不通过** 时，进入循环治理。循环治理控制 Full Loop Engineering 的收敛过程，确保：

1. 循环有限步内结束（不陷入死循环）
2. 每轮循环都有有效进展
3. 达到上限后自动升级处理

```
Evaluation → 不通过
    ↓
┌─ re-execute ──────────────────────────────┐
│  同层重试：修复具体问题，不改变设计方案     │
│  适用：check-types error、lint error、文件  │
│  遗漏等可量化问题                           │
└────────────────────────────────────────────┘
    ↓ 连续重试无效
┌─ re-plan ──────────────────────────────────┐
│  升级重来：修改设计方案、调整约束/策略      │
│  适用：架构缺陷、设计矛盾、重复性问题        │
└────────────────────────────────────────────┘
    ↓ 规划无效
┌─ 人工介入 ──────────────────────────────────┐
│  上报路由协调人工处理                        │
│  适用：3 次循环无进展、安全漏洞无法修复      │
└─────────────────────────────────────────────┘
```

---

## Constraint（硬约束）

### 循环计数

- 同一任务的 **re-execute 最大次数**：3 次
- 同一任务的 **re-plan 最大次数**：2 次
- 同一个 Evaluation 任务从首次执行到最终通过的 **总循环上限**：5 次（re-execute + re-plan 合计）
- 达到上限仍未通过 → 强制升级到人工介入

### 循环范围

- 每次循环均限定在当前任务范围内，不得跨越到新任务
- 依赖链任务中某一步失败，只循环该步骤，不重跑已通过的步骤
- 并行子任务中某一条失败，只循环失败子任务，不影响其他已完成子任务

### 循环变化检测

- re-execute 第 2 次时，必须对比第 1 次的失败原因。如果**完全相同**，禁止第 2 次 re-execute，强制升级到 re-plan
- re-plan 第 2 次时，必须对比第 1 次的方案。如果方案没有实质性改变（仅措辞调整），禁止第 2 次 re-plan，强制升级到人工介入

### 每次循环的最低产出

- 每次 re-execute 后的产出必须与前一次**有可量化的改进**（减少的 lint error 数、修复的文件数等）
- 每次 re-plan 后的方案必须与前一次**有实质性差异**（不同的架构方式、不同的工具选择）
- 无实质改进的循环视为"无效循环"，直接升级

### 工具调用去重

- 同一轮循环中，**禁止使用完全相同的参数重复调用同一个工具**
- 每次工具调用后，记录 `工具名 + 参数摘要（取前 100 字符的哈希）` 到循环状态中
- 下次调用前，先检查是否已存在相同记录
- 如果检测到重复调用意图：
  - 第 1 次重复 → 跳过调用，直接复用上次结果
  - 第 2 次重复 → 视为"无效循环"，升级处理
- 依赖链中不同步骤间的同一工具调用不适用此约束（前一步写入 → 下一步读取是正常流程）

### 静默失败检测

- 每次工具调用后，**必须验证返回结果的有效性**
- 以下情况视为静默失败，不得继续向下执行：
  - 工具返回空字符串或空对象（`{}`）
  - 工具返回 `null` 或 `undefined`
  - 工具返回异常状态码但无错误信息
  - 工具返回了预期结构但不包含实际内容（如空数组、空文件内容）
- 检测到静默失败时的处理流程：
  ```
  检测到静默失败
    → 重新调用一次工具（最多 1 次重试）
    → 仍然静默失败？→ 上报异常，触发 re-execute
    → 连续 2 轮静默失败？→ 升级到 re-plan
  ```
- 依赖链上下文中传递空值也按静默失败处理

### 重试退避

- 连续三次或以上的工具调用失败，必须使用指数退避：
  ```
  第 1 次失败 → 立即重试（不退避）
  第 2 次失败 → 等待 1 秒后重试
  第 3 次失败 → 等待 2 秒后重试
  第 4 次失败 → 等待 4 秒后重试
  第 N 次失败 → 等待 2^(N-2) 秒后重试（上限 60 秒）
  ```
- 每次退避等待需加入 ±20% 的随机抖动（jitter），避免"惊群效应"
- 退避期间应标记该工具为"不稳定"状态，本轮循环中优先考虑替代方案
- 如果退避超 60 秒仍失败 → 视为该工具当前不可用，按 MCP/Skill 降级策略处理

### 成本 / 预算约束

- 单次循环的总工具调用次数上限：**30 次**（含所有工具的累计调用）
- 达到 30 次后，无论是否还有可用步数，强制触发 re-plan 或人工介入
- AI 应记录每次工具的消耗情况（调用次数、结果类型），在状态中追踪
- 如果某次循环消耗明显超出预期（如同一工具调用超 5 次），暂停并输出成本预警

## Heuristic（启发式）

### 循环执行流程

```
Evaluation 不通过
    ↓
检查失败类型：

check-types error / lint error / 文件遗漏
    → re-execute（同层重试）
    → 修复 → 重新评估
    → 通过？→ 结束
    → 不通过？→ 检查失败原因与上次是否相同
        → 不同？→ 再试（计数 +1）
        → 相同？→ 升级到 re-plan

架构缺陷 / 设计矛盾 / 重复问题
    → re-plan（升级重来）
    → 修改方案 → 重新路由 → 重新执行 → 重新评估
    → 通过？→ 结束
    → 不通过？→ 检查方案是否实质性改变
        → 有改变？→ 再试（计数 +1）
        → 无改变？→ 升级到人工介入

安全隐患 / 未知错误 / 3 次无效
    → 直接人工介入
    → 上报路由协调
```

### 循环状态记录

每轮循环必须记录以下状态，供对比和决策：

```json
{
  "cycle": 1,
  "type": "re-execute",
  "failed_items": [
    {
      "check": "pnpm check-types",
      "error_count": 3,
      "error_location": "src/components/Form.vue:45"
    }
  ],
  "previous_failure_overlap": false,
  "improvement_metric": "-50% error"
}
```

### re-execute → re-plan 切换判断

使用以下判断表决定是否切换：

| 当前状态                                     | 建议动作               |
| -------------------------------------------- | ---------------------- |
| 第 1 次 re-execute 失败                      | 允许第 2 次 re-execute |
| 第 2 次 re-execute 失败（新问题）            | 允许第 3 次 re-execute |
| 第 2 次 re-execute 失败（与第 1 次相同问题） | **切换到 re-plan**     |
| 第 3 次 re-execute 失败                      | **切换到 re-plan**     |
| 第 1 次 re-plan 失败（新方案）               | 允许第 2 次 re-plan    |
| 第 1 次 re-plan 失败（方案无实质改变）       | **切换到人工介入**     |
| 第 2 次 re-plan 失败                         | **切换到人工介入**     |

### 无效循环检测方式

- 对比失败项目列表：用 `grep` 或字符串匹配检查前次失败的文件路径和错误信息
- 对比评估报告：读取前次评估报告的输出，逐项对比
- 如果完全一致 → 判定为无效循环
- 如果部分一致但上次修复未生效 → 判定为新问题（有效循环）

### 语义循环检测

硬性的"完全相同"检测不够，需要识别"换了说法但本质相同"的语义循环：

| 语义循环模式                                                       | 检测方式                                       | 判定     |
| ------------------------------------------------------------------ | ---------------------------------------------- | -------- |
| 同一文件同一行报同一类错误                                         | 检查失败文件路径 + 行号范围是否重叠            | 语义循环 |
| 相同工具名 + 参数结构相似（如换了一个无关参数）                    | 检查工具名 + 参数 key 集合是否一致             | 语义循环 |
| 修复方案描述不同但实际做了同一件事                                 | 对比前后两次 Read 同一文件，检查 diff 是否一致 | 语义循环 |
| 连续 2 次失败报错信息包含相同的模板（如 `Cannot find module '*'`） | 用正则提取错误模板，去变量后对比               | 语义循环 |
| 重新打开已关闭的问题                                               | 检查文件路径在上次失败中出现过                 | 语义循环 |

检测到语义循环时的处理方式：

- 第 1 次语义循环 → 记录"疑似语义循环"，允许一次差异化尝试（换完全不同的方案）
- 第 2 次语义循环 → 判定为无效循环，强制升级到 re-plan（不等待次数上限）
- 语义循环的检测优先级高于普通失败对比——即如果同时有硬性不同但语义相同，按语义循环处理

### 上下文压缩策略

长循环中上下文会持续膨胀，导致模型降智和成本飙升。必须在循环过程中主动压缩：

```
每 2 次 re-execute 或每 5 次工具调用后（以先到为准）：
    → 读取循环状态记录中的 summary 字段
    → 将历史工具调用详情压缩为"已做的尝试及其结果"摘要
    → 丢弃历史中不再需要的原始输出（保留摘要版本）
    → 保留以下不可压缩的信息：
       - 原始任务目标
       - 已完成的步骤清单
       - 当前失败的类型和位置
```

可压缩的内容：

| 可以压缩                     | 不可压缩           |
| ---------------------------- | ------------------ |
| 各工具调用的完整返回体       | 原始任务目标       |
| 中间推理步骤的完整思维链     | 失败错误信息原文   |
| 依赖链中上一步的完整 context | 已完成的步骤清单   |
| 评估报告的详细输出           | 人工介入的历史记录 |

压缩后保留 3 个关键段：

```
[目标] 原始任务目标（保持不变）
[进度] 当前已完成的步骤清单 + 当前循环次数
[失败] 本轮失败的检查项 + 错误位置 + 已尝试的修复方案摘要
```

### 目标锚定机制

每次进入新循环迭代前，必须先做目标一致性检查：

```
① 读取原始任务目标（来自 workflow 或 execution-plan）
② 对比当前正在执行的内容
③ 偏差度评估：
   - 0% → 完全一致，继续
   - < 20% → 有偏差但可接受，修正后继续
   - ≥ 20% → 已脱离原始目标，暂停并上报
```

偏差度评估参考：

| 表现                                | 偏差度 |
| ----------------------------------- | ------ |
| 仍在做原始任务，只是换了实现方式    | 0%     |
| 做了原始任务 + 额外做了附近的小优化 | < 20%  |
| 原始任务没做完就去解决依赖项的问题  | ≥ 20%  |
| 原始任务被搁置，转向了另一个方向    | ≥ 50%  |

检测到目标漂移后：

- 偏差度 < 20% → 放弃额外工作，回到原始任务
- 偏差度 ≥ 20% → 暂停当前工作，输出"目标漂移报告"到路径，触发 re-plan
- 偏差度 ≥ 50% → 直接上报人工介入

### 状态持久化方案

循环状态必须在每一步持久化到可复现的路径，确保中断后可恢复：

```
每完成一次 re-execute 或 re-plan 后：
    → 写入 .trae/loop-state/{task-id}-cycle-{n}.json
    → 文件内容包括循环状态 JSON（循环次数、类型、失败项列表）
    → 保留所有历史状态文件，不覆盖（方便追溯）
```

状态文件命名规则：

```
.trae/loop-state/{task-hash}-cycle-001.json
.trae/loop-state/{task-hash}-cycle-002.json
```

如果循环被中断（进程重启、超时等）：

```
① 扫描 .trae/loop-state/ 下是否存在该 task 的最近状态文件
② 读取最新循环次数 + 失败项
③ 从该次数继续，不从头开始
④ 如果状态文件损坏或格式不正确 → 从头开始（安全降级）
```

## Policy（决策策略）

### 循环次数上限汇总

| 指标                                | 上限 | 超出后         |
| ----------------------------------- | ---- | -------------- |
| re-execute 次数                     | 3    | 强制 re-plan   |
| re-plan 次数                        | 2    | 强制人工介入   |
| 总循环数（含 re-execute + re-plan） | 5    | 强制人工介入   |
| 无效循环次数                        | 1    | 强制升级下一级 |

### 失败分类 → 动作映射

| 失败类型               | 首轮动作                   | 后续升级路径                       |
| ---------------------- | -------------------------- | ---------------------------------- |
| check-types error      | re-execute                 | re-execute → re-plan → 人工        |
| lint error             | re-execute                 | re-execute → re-plan → 人工        |
| 计划文件缺失           | re-execute                 | re-execute → re-plan               |
| 计划外文件             | re-execute                 | re-execute                         |
| Migration 无 down      | re-execute                 | re-execute                         |
| 模块注册遗漏           | re-execute                 | re-execute                         |
| Breaking change 未标注 | re-execute                 | re-execute                         |
| API key 硬编码         | **人工介入**（直接）       | 安全事件                           |
| 性能退化 > 10%         | re-plan                    | re-plan → 人工                     |
| 安全审计 high 未修复   | **人工介入**（直接）       | 安全事件                           |
| 依赖链 context 丢失    | re-execute（重跑该步骤）   | re-execute → re-plan               |
| 语义错误               | re-plan                    | re-plan → 人工                     |
| 工具调用重复           | 跳过调用，复用上次结果     | 第 2 次重复 → 无效循环升级         |
| 静默失败（工具返回空） | 重新调用 1 次              | 连续 2 轮 → re-plan                |
| 退避超 60 秒           | 标记工具不可用，走替代方案 | 替代方案失败 → re-plan             |
| 调用次数超 30          | 成本熔断 → 强制 re-plan    | 人工介入                           |
| 语义循环               | 允许一次差异化尝试         | 第 2 次 → 直接 re-plan（不等上限） |
| 目标漂移 ≥ 20%         | 暂停，触发 re-plan         | ≥ 50% → 人工介入                   |
| 循环中断（状态丢失）   | 从状态文件恢复             | 无状态文件 → 从头开始              |

### 人工介入的触发条件

以下任一条件满足，强制触发人工介入：

1. re-execute 达到 3 次上限仍未通过
2. re-plan 达到 2 次上限仍未通过
3. 总循环数达到 5 次上限
4. 检测到无效循环（连续两次相同失败）
5. 安全相关失败（硬编码密钥、SQL 注入等）
6. 评估发现无法在当前治理框架内解决的问题

### 人工介入后的流程

```
人工介入触发
    ↓
输出完整报告：
  - 当前循环次数和类型
  - 失败项目清单（含历史对比）
  - 已尝试的修复方案
  - 建议下一步方向
    ↓
上报到路由 → 等待人工决策
    ↓
人工决策结果：
  ① 认可方案 → 手动批准 → 继续执行
  ② 否定方案 → 提供新方向 → 重新路由
  ③ 取消任务 → 标记为不可行
```

### 循环异常降级总表

| 异常场景                       | 处理方式                     |
| ------------------------------ | ---------------------------- |
| 同问题连续重试 3 次            | 升级 re-plan                 |
| re-plan 出相同方案             | 升级人工介入                 |
| 安全漏洞循环中未修             | 立即升级人工介入             |
| 总循环超 5 次                  | 强制终止，人工介入           |
| 依赖链中间步骤成功但下游重新跑 | 不允许，保持中间步骤缓存     |
| 并行子任务一失败拖慢其他       | 独立循环，不阻塞并行任务     |
| AI 自主在循环中扩大任务范围    | 阻断并回归原任务范围         |
| 工具调用重复（第 2 次）        | 判定无效循环，升级 re-plan   |
| 静默失败连续 2 轮              | 升级 re-plan                 |
| 退避超 60 秒无替代方案         | 升级 re-plan                 |
| 工具调用次数超 30              | 成本熔断，强制 re-plan       |
| 语义循环第 2 次                | 直接 re-plan（跳过次数上限） |
| 目标漂移 ≥ 20%                 | 暂停，触发 re-plan           |
| 目标漂移 ≥ 50%                 | 直接人工介入                 |
| 循环中断且有状态文件           | 从最近状态文件恢复           |
| 循环中断且无状态文件           | 从头开始（安全降级）         |
