# Constraint — 评估阶段通用硬约束

## 评估前约束

- 评估前必须先读取对应领域的 `execution-engine/{domain}/` 了解执行阶段的约束/启发/策略
- 评估前必须先读取对应 `workflows/{domain}/{task-type}.md` 的完成检查清单
- 评估前必须先读取 `execution-plan/{domain}/` 确认规划期的承诺与约束
- 评估对象必须是执行阶段的实际产出文件，不得基于记忆或预期

## 验证门禁

- 类型检查 / lint / format / build — 按各领域的具体命令执行
- 任一门禁未通过，不得输出来完成结论
- 各领域的门禁标准以对应 `evaluation/{domain}/constraint.md` 为准

## 完整性检查

- 计划中列出的新建文件必须真实存在
- 计划中列出的修改文件必须真实被修改
- 计划中列出的删除文件必须已删除或标记废弃
- 计划中标注的"只读"文件不得有变更
- 评估范围内不得存在未列入计划的新建文件（临时辅助文件除外）
- 依赖链多步骤任务的上下文交接内容必须可追溯

## 范围约束

- 评估仅对当前任务的产出负责，不评估无关代码
- 发现任务范围外的 Bug 时，记录但不阻断，标注"非本次任务范围"
- 发现任务范围外的安全隐患或敏感数据泄露时，记录并上报路由
- 评估结论必须基于可复现的证据（命令输出、文件内容），不得凭感觉判断

## 输出约束

- 评估结论必须输出结构化报告，包含：领域、任务类型、评估项清单、每项通过/失败状态、失败原因、证据来源
- 评估报告必须是可读的（他人可阅读并理解）
- 结论有三种：**通过** / **有条件通过**（附问题清单）/ **不通过**（附原因并触发 re-plan 或 re-execute）
- 输出格式可被 evaluation 的下一环节（人工介入/自迭代）直接消费
