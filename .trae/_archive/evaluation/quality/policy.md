# Policy — 质量评估阶段决策策略

## 通过/不通过决策

| 评估结果                              | 结论           | 后续动作                     |
| ------------------------------------- | -------------- | ---------------------------- |
| 测试全通过 + 无 high CVE + 性能无退化 | **通过**       | 标记完成，输出报告           |
| 测试全通过 + 有 moderate 问题         | **有条件通过** | 记录问题，标记完成           |
| 测试失败                              | **不通过**     | 触发 re-execute              |
| 安全审计有 high 未修复                | **不通过**     | 触发 re-plan（优先修复安全） |
| 性能退化 > 10%                        | **不通过**     | 触发 re-plan（优化后重测）   |
| 审查输出不完整                        | **不通过**     | 触发 re-execute（补全输出）  |

## 阻断性 vs 非阻断性判定

| 发现                           | 分类     | 理由             |
| ------------------------------ | -------- | ---------------- |
| 测试用例失败                   | 阻断性   | 功能不可靠       |
| 测试覆盖不足（新增代码缺测试） | 非阻断性 | 记录待补充       |
| high CVE                       | 阻断性   | 安全隐患         |
| 性能退化 > 10%                 | 阻断性   | 用户体验下降     |
| 性能退化 5-10%                 | 非阻断性 | 记录，可后续优化 |
| 审查发现阻断性缺陷             | 阻断性   | 必须修复         |
| 审查发现编码风格问题           | 非阻断性 | 记录即可         |
| API 测试全部通过               | 通过     | -                |
| API 测试边界条件失败           | 非阻断性 | 记录待修复       |

## 安全问题分级

| 等级     | 定义                     | 处理时限       |
| -------- | ------------------------ | -------------- |
| critical | 远程代码执行、未授权访问 | 立即修复       |
| high     | 敏感数据泄露、权限绕过   | 本次任务内修复 |
| moderate | SQL 注入未输入过滤       | 本次或下次排期 |
| low      | 信息泄露（暴露版本号等） | 记录，持续改进 |

## 性能基线对比

- P50 退化 < 5% → 正常
- P50 退化 5-10% → 记录，排期优化
- P50 退化 > 10% → 不通过，触发 re-plan
- P95/P99 退化 > 10% → 不通过，优先排查

## 评估结果影响全链路

```
QA 评估不通过
→ 检查修复前提（是否需要 re-plan 还是可 re-execute）
→ 如果只是单文件问题 → re-execute
→ 如果涉及到架构/设计变更 → re-plan（回路由重走）
→ 如果安全审计 high 未修复 → 优先阻断，不进入下一阶段
```
