# Constraint — 评估阶段通用硬约束

## 评估前约束

- 评估前必须先读取对应领域的 `execution-engine/{domain}/` 了解执行阶段的约束/启发/策略
- 评估前必须先读取对应 `workflows/{domain}/{task-type}.md` 的完成检查清单
- 评估前必须先读取 `execution-plan/{domain}/` 确认规划期的承诺与约束
- 评估对象必须是执行阶段的实际产出文件，不得基于记忆或预期

## 验证门禁

- 类型检查 / lint / format / build — 按各领域的具体命令执行
- 任一门禁未通过，不得输出来完成结论
- 各领域的门禁标准以对应 `evaluation/{domain}/constraint.md` 为准

## 完整性检查

- 计划中列出的新建文件必须真实存在
- 计划中列出的修改文件必须真实被修改
- 计划中列出的删除文件必须已删除或标记废弃
- 计划中标注的"只读"文件不得有变更
- 评估范围内不得存在未列入计划的新建文件（临时辅助文件除外）
- 依赖链多步骤任务的上下文交接内容必须可追溯

## 静默成功检测

工具调用了不等于执行成功了。以下 10 种情况判定为**静默失败**，不得输出来完成结论：

| 检测项              | 判定条件                                             | 动作                            |
| ------------------- | ---------------------------------------------------- | ------------------------------- |
| **文件零字节**      | 新建的文件大小为 0 bytes                             | 标记 FAIL，要求重新创建         |
| **文件内容无变更**  | 修改后文件内容和修改前完全一致                       | 标记 FAIL，要求确认修改范围     |
| **工具返回空**      | MCP/tool 调用返回空结果或仅含 whitespace             | 标记 FAIL，重试一次，仍空则升级 |
| **命令输出空**      | `RunCommand` 执行成功(exit 0)但 stdout/stderr 均为空 | 标记 WARN，确认是否预期行为     |
| **Git 无 diff**     | git diff 显示无任何变更                              | 标记 FAIL，确认修改未生效原因   |
| **MCP 静默**        | MCP 返回 success 状态但 payload 为空或缺少关键字段   | 标记 FAIL，要求重新调用         |
| **部分成功**        | 多文件操作中仅部分文件写入成功，未报告失败文件       | 标记 FAIL，要求补全失败文件     |
| **缓存命中掩盖**    | turbo cache 命中导致任务实际未执行但返回成功         | 标记 FAIL，要求 `--force` 重跑  |
| **权限降级返回空**  | Supabase RLS 策略返回空集而非报错（非 404）          | 标记 FAIL，要求检查 RLS 策略    |
| **deprecated 警告** | 工具返回 deprecated 警告但执行成功，可能下次版本失效 | 标记 WARN，记录到经验数据       |

检测规则：

1. **每项工具调用/文件操作后必查** — 不得跳过静默检测直接进入下一步
2. **连续 2 次静默失败** → 升级到循环治理，按 `loop-governance.md` 处理
3. **命令有 stdout 但无文件变更** → 属于"执行了但没产出"，标记 FAIL
4. **静默失败记录**必须写入评估报告，供 evolution 收集分析

## 范围约束

- 评估仅对当前任务的产出负责，不评估无关代码
- 发现任务范围外的 Bug 时，记录但不阻断，标注"非本次任务范围"
- 发现任务范围外的安全隐患或敏感数据泄露时，记录并上报路由
- 评估结论必须基于可复现的证据（命令输出、文件内容），不得凭感觉判断

## 输出约束

- 评估结论必须输出结构化报告，包含：领域、任务类型、评估项清单、每项通过/失败状态、失败原因、证据来源
- 评估报告必须是可读的（他人可阅读并理解）
- 结论有三种：**通过** / **有条件通过**（附问题清单）/ **不通过**（附原因并触发 re-plan 或 re-execute）
- 输出格式可被 evaluation 的下一环节（人工介入/自迭代）直接消费
