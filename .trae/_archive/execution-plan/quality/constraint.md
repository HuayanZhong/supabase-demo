# Constraint — 质量规划硬约束

## 测试约束

- 新增测试不得破坏现有测试（运行现有测试通过后再添加）
- api-test-pro agent 只做测试验证，不得修改业务代码（不做 Edit/Write 操作）
- 性能分析必须基于工具产出的数据（Lighthouse、Clinic.js），不得凭感觉判断

## 审查约束

- 代码审查必须基于实际代码变更，不得审查未读的代码
- 安全审计必须检查依赖中的已知漏洞（通过 `npm audit` 或类似工具）
- 审查意见必须具体到代码行和问题类型，不得泛泛而谈

## 安全约束

- 涉及密码、密钥、Token 的代码必须标识为需要额外审查
- 安全修复必须标注 CVE 编号或漏洞来源

## 范围约束

- 质量任务只做分析和验证，不做功能开发
- 审查中发现业务逻辑问题→记录并上报路由，不直接修改
