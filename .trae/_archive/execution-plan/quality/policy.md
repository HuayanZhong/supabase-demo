# Policy — 质量规划决策策略

## 测试决策

| 场景       | 策略                                             |
| ---------- | ------------------------------------------------ |
| 工具函数   | 纯函数测试，覆盖正常/边界/异常 3 类输入          |
| API 端点   | 测试状态码+响应结构，覆盖成功/失败/鉴权 3 种情况 |
| UI 组件    | 视觉回归测试由手动确认，automatic 覆盖交互逻辑   |
| 数据库查询 | 验证 SQL 执行结果符合预期                        |

## 问题分级决策

| 严重级别 | 定义                           | 处理方式               |
| -------- | ------------------------------ | ---------------------- |
| 阻断性   | 功能不可用、数据安全风险       | 必须修复，否则不可发布 |
| 主要     | 功能受损但可绕过、性能显著下降 | 建议本轮修复           |
| 次要     | 轻微 UI 偏差、非关键路径问题   | 可记录到后续迭代       |
| 建议     | 代码风格、可读性改进           | 仅记录，不阻塞         |

## 性能优化决策

| 场景       | 策略                                |
| ---------- | ----------------------------------- |
| 首次加载慢 | 代码分割 + 懒加载 + 缓存策略        |
| 运行时卡顿 | 减少重渲染 + 虚拟列表 + 计算缓存    |
| API 响应慢 | 加索引 + 优化查询 + 添加缓存层      |
| 构建慢     | 检查依赖 + 优化 turbo.json pipeline |

## 安全修复优先级

| 风险等级           | 修复期限      |
| ------------------ | ------------- |
| 高危（CVSS ≥ 9）   | 立即修复      |
| 中危（CVSS 7-8.9） | 24 小时内修复 |
| 低危（CVSS < 7）   | 下个迭代修复  |

## 审查优先级

| 审查内容         | 优先级 |
| ---------------- | ------ |
| 安全相关变更     | 最高   |
| 数据库 Migration | 高     |
| 公共 API 变更    | 高     |
| 新功能代码       | 中     |
| 重构/样式变更    | 低     |
