# Policy — DevOps 规划决策策略

## CI 配置决策

| 场景         | 策略                                |
| ------------ | ----------------------------------- |
| 新增 CI 流程 | 参考现有 workflow 文件结构和命名    |
| 修改触发条件 | 确认不会导致重复触发或漏触发        |
| 添加缓存     | 优先用 `actions/cache` + pnpm store |
| 并行 Job     | 无依赖的 Job 可并行，有依赖的串行   |

## 依赖升级决策

| 场景         | 策略                                   |
| ------------ | -------------------------------------- |
| 补丁版本升级 | 直接升级，跑一次 `pnpm install` 验证   |
| 次版本升级   | 先读 changelog，确认无 breaking change |
| 主版本升级   | 标注为高风险操作，建议单独安排验证周期 |
| 新增依赖     | 先确认是否已有同类依赖，避免重复       |

## 部署决策

| 场景             | 策略                               |
| ---------------- | ---------------------------------- |
| 首次部署         | 先确认托管平台配置、域名、SSL 证书 |
| 增量部署         | 只部署变更的 app，不影响其他服务   |
| 回滚部署         | 确认回滚方案后再执行部署           |
| 数据库 Migration | 部署前确认 migration 脚本已就绪    |

## 环境配置决策

| 场景         | 策略                                 |
| ------------ | ------------------------------------ |
| 新增环境变量 | 同时在 `.env.example` 和部署平台配置 |
| 修改环境变量 | 确认所有使用方同步更新               |
| 删除环境变量 | 先搜索代码中是否有引用，确认无残留   |
