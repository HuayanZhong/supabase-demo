# Policy — AI 规划决策策略

## 模型接入决策

| 场景              | 策略                                         |
| ----------------- | -------------------------------------------- |
| 接入标准 Chat API | 直接调用 REST API（OpenAI 兼容格式）         |
| 需要流式输出      | 使用 SSE / Server-Sent Events                |
| 需要嵌入向量      | 先确认 pgvector 是否启用，再用 Supabase 存储 |
| 需要多轮对话      | 设计会话上下文管理（存 session_id → 查历史） |

## 错误处理决策

| 场景             | 策略                                    |
| ---------------- | --------------------------------------- |
| 模型返回超时     | 重试 1 次，失败则用缓存或默认值兜底     |
| 模型返回格式异常 | 校验结构 + 提示用户重试，不展示原始错误 |
| API Key 无效     | 提示环境变量配置问题，不暴露 Key 值     |
| 频率限制         | 退避重试 + 提示用户稍后再试             |

## 数据策略

| 场景         | 策略                                |
| ------------ | ----------------------------------- |
| 涉及用户隐私 | 不发送到第三方模型，考虑本地模型    |
| 需要上下文   | 只传必要的历史记录，控制 Token 长度 |
| 需要知识库   | 用 RAG 方案（embedding + 向量检索） |
| 需要实时数据 | 通过 Tool Calling 调用内部 API 获取 |

## 架构决策

| 场景         | 策略                               |
| ------------ | ---------------------------------- |
| AI 逻辑轻量  | 直接在 API Route 中调用            |
| AI 逻辑复杂  | 抽取独立 Service + Module          |
| 需要后台任务 | 使用 Queue / Edge Functions        |
| 多模型切换   | 抽象 Model Provider 接口，支持切换 |
