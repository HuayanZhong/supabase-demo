# Policy — 前端执行阶段决策策略

## MCP vs Skill vs 直接工具决策

| 需要的信息           | 首选                                       | 何时降级                                |
| -------------------- | ------------------------------------------ | --------------------------------------- |
| Nuxt UI 组件是否存在 | `nuxt-ui MCP → search-components`          | MCP 不可用时 → `skill/nuxt-ui` 查询文档 |
| 组件具体 API 参数    | `nuxt-ui MCP → get-component`              | MCP 不可用时 → `skill/nuxt-ui`          |
| 组件使用示例         | `nuxt-ui MCP → get-example`                | 无示例时 → 参考项目中已有用法           |
| 图标名称确认         | `nuxt-ui MCP → search-icons`               | 无降级方案，不得猜测                    |
| Supabase 库表诊断    | `supabase MCP → list_tables / execute_sql` | MCP 不可用时 → `skill/supabase`         |
| 项目代码搜索         | `Grep` 或 `SearchCodebase`                 | 语义搜索优先于正则                      |
| 项目文件查找         | `Glob`                                     | 路径已知时直接用 `Read`                 |

**原则**：MCP 最精确（活的 API 文档）→ skill 次之（静态知识库）→ 直接工具兜底（代码级操作）。三者覆盖不同粒度，不存在完全替代关系。

## 执行中断与恢复决策

| 场景                 | 策略                                        |
| -------------------- | ------------------------------------------- |
| 类型检查失败         | 阻断性错误 → 停止当前步骤，修复类型后再继续 |
| lint 失败            | 非阻断性错误 → 记录问题，完成后统一修复     |
| 语法错误             | 阻断性错误 → 立即修复                       |
| MCP 调用失败         | 降级到 skill（如上表），记录降级原因        |
| 文件已存在（创建时） | 先 Read 现有内容，确认是追加/覆写/跳过      |
| 目标路径不存在       | 先创建目录，再写入文件                      |

## Write vs SearchReplace 决策

| 场景                     | 推荐方式             | 理由               |
| ------------------------ | -------------------- | ------------------ |
| 全新文件                 | `Write`              | 没有已有内容可参考 |
| 修改文件中的连续代码块   | `SearchReplace`      | 保留周围上下文     |
| 修改文件中的多处分散代码 | 多次 `SearchReplace` | 每次只改一个逻辑点 |
| 文件大部分内容需要重写   | 分段 `SearchReplace` | 避免误删未改部分   |
| 文件内容少且全改         | `Write` 覆写         | 比多次替换更清晰   |

**原则**：能局部改就不要全量写，减少意外变更的风险。

## 执行顺序决策

| 情形                 | 策略                                                     |
| -------------------- | -------------------------------------------------------- |
| 有多个文件要修改     | 按依赖顺序：类型定义 → composable → 组件 → 页面          |
| 同时有新建和修改     | 先新建（不依赖已有代码），再修改（可能需要引用新建内容） |
| 同时有前端和后端变更 | 这不是单领域任务，暂停并上报路由分配依赖链               |
| 同一个文件多次修改   | 每次修改后确认正确，再进入下一次修改                     |

## 错误分级与处理

```
阻断性错误（必须立即处理）：
├── 类型错误（check-types 失败）
├── 语法错误（编辑器诊断报红）
├── 依赖缺失（import 找不到模块）
└── 文件写入失败（权限/磁盘）

非阻断性错误（记录，完成后处理）：
├── lint 警告（非 error 级别）
├── 格式问题（oxfmt 不一致）
└── 未使用的变量/import
```

遇到非阻断性错误堆积过多（超过 5 条）时，升格为优先处理。

## 资源冲突决策

| 冲突场景                        | 策略                                                     |
| ------------------------------- | -------------------------------------------------------- |
| 多个 MCP 都可满足需求           | 选最专精的（nuxt-ui MCP 用于 UI，supabase MCP 用于数据） |
| skill 和 MCP 信息矛盾           | MCP 优先（实时数据 > 静态知识）                          |
| workflow 和 execution-plan 矛盾 | execution-plan 优先（约束 > 流程建议）                   |
| 用户直接指令与规则矛盾          | 用户指令优先，但标注风险                                 |
