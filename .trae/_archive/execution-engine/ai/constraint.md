# Constraint — AI 执行阶段硬约束

## 执行前约束

- 开始执行前，必须先读取对应 `workflows/ai/{type}.md` 了解执行步骤
- 开始执行前，必须先读取 `execution-plan/ai/` 了解规划约束/启发/策略
- 开始执行前，必须通过 `runtime/{domain}/router.md` 确认分配的资源（rules、skills、MCP）

## AI 配置与密钥约束

- 执行涉及外部 AI API 的任务前，必须确认 API key 和模型参数已正确配置
- 不得将 API key、token、密码或任何凭证硬编码到源码中
- API key 必须从环境变量或项目已有的 secret 机制读取
- 修改 AI 配置前，必须先 Read 当前配置内容，确认无误后再修改
- 配置变更后必须验证配置语法正确性

## 资源调用约束

- 涉及 web 搜索/实时数据获取时，必须使用 `tavily_search MCP` 而非通用工具
- 调用 MCP 前必须先通过 Read 读取对应的工具描述文件，确认参数后再调用
- 涉及 AI 集成时，优先使用已有的 `ai-integration-engineer` skill
- 不得直接调用未在 workflow 资源表中列出的 MCP 或 skill

## 执行过程约束

- 执行必须严格按 `workflows/ai/{type}.md` 的步骤顺序进行，不得跳过
- 每一步完成后必须确认输出是否符合预期，确认后再进入下一步
- 执行过程中发现计划外的文件需要修改时，必须先上报再修改
- 涉及创建文件时，必须先确认目标路径是否存在，不存在则先创建目录
- 涉及修改文件时，必须先 Read 当前内容再修改，不得凭记忆重写
- 不得在单个工具调用中写入超过 200 行的文件

## AI 功能开发约束

- AI 功能变更必须考虑外部 API 的 rate limit，实现指数退避重试机制
- 所有外部 API 调用必须包含完善的错误处理（网络错误、超时、认证失败）
- 流式响应（streaming）必须处理连接中断和超时异常
- 非流式响应必须处理超时和部分响应场景
- AI 生成的代码/内容在集成前必须经过质量验证
- 不得修改分配范围之外的核心业务逻辑

## 验证约束

- 所有涉及 AI 集成的变更，必须通过 mock 测试验证
- 所有涉及 RAG 的变更，必须验证 embedding pipeline 和检索质量
- 涉及配置变更的任务，必须验证配置语法正确性
- 所有变更必须通过 lint 和格式化验证

## 错误处理约束

- 执行中遇到错误时，必须停止当前步骤并记录错误信息，不得尝试绕过
- 错误分类：
  - **阻断性错误**（类型错误、语法错误、依赖缺失、API key 缺失）— 必须修复后才能继续
  - **非阻断性错误**（lint 警告、格式问题）— 记录但可继续，完成后统一修复
- 同一错误连续 2 次修复尝试仍失败时，必须暂停执行并上报
- 修改必须可回滚（通过 git），破坏性变更前必须执行 `git stash` 或提交当前工作区
